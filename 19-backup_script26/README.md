
# Практика написання скриптів

---

## Створювати правильну структуру скрипта, що відповідає стандартам production

- Структура скрипта — це не лише для краси, а для того, щоб код був зрозумілим, підтримуваним і зручним для діагностики.
- Професійний скрипт має чіткий поділ на:
  - Заголовок (метадані)
  - Імпорти/оголошення змінних
  - Функції
  - Основний блок виконання
  - Обробку помилок і логування
  
---

## Обробляти вхідні параметри, роблячи інструменти універсальними

- Скрипти повинні приймати параметри командного рядка (наприклад, `-v`, `--backup-dir=/path/to/dir`).
- Використовуйте `getopts` або `getopt` для обробки опцій.
- Це робить скрипт гнучким і зручним для різних сценаріїв використання.

---

## Реалізовувати логіку бекапу та деплою з реальними перевірками

- Перевіряйте існування директорій, прав доступу, вільного місця.
- Виконуйте бекап з перевіркою цілісності (наприклад, перевірка хеш-сум).
- В деплої — перевіряйте успішність команд, наявність потрібних файлів, коректність версій.

---

## Впроваджувати професійне логування та систему сповіщень

- Логування — це запис у файл або вивід у консоль з рівнями (INFO, WARN, ERROR).
- Сповіщення — надсилання email або повідомлень у системи моніторингу при помилках.
- Використовуйте функції для логування, щоб уникнути дублювання коду.

---

## Безпечно інтегрувати скрипти з cron, уникаючи типових помилок середовища

- Вказуйте повні шляхи до команд.
- Встановлюйте змінні середовища (`PATH`, `LANG`).
- Перенаправляйте вивід і помилки у лог-файли.
- Використовуйте `MAILTO` для отримання сповіщень.
- Перевіряйте, що скрипт виконується у потрібному контексті (наприклад, перевірка поточного користувача).

---

# Анатомія професійного скрипта

---

## Чому структура важливіша за команди?

- Команди — це лише інструменти.
- Структура визначає, як легко підтримувати, розширювати і тестувати скрипт.
- Добре структурований скрипт легше читати, розуміти і змінювати.
- Це особливо важливо у production, де помилки дорого коштують.

---

## Шебанг (Shebang) — вибір інтерпретатора

- Перший рядок скрипта задає, яким інтерпретатором він буде виконуватись.
- Найпоширеніші варіанти:

```bash
#!/bin/bash
```

або для більш портативного варіанту:

```bash
#!/usr/bin/env bash
```

- Другий варіант шукає інтерпретатор у `PATH` і кращий для систем з різними шляхами.

---

## Заголовок та метадані (Docstring)

- Включаєте інформацію про:

  - Назву скрипта
  - Опис призначення
  - Авторів
  - Дату створення/оновлення
  - Використання (параметри)
  - Приклади запуску

---

**Приклад:**

```bash
#!/bin/bash
# backup.sh - Скрипт для резервного копіювання директорії
# Автор: Ivan Ivanov
# Дата: 2026-06-01
# Використання: ./backup.sh -s /source/dir -d /backup/dir
# Приклад: ./backup.sh -s /home/user -d /mnt/backup
```

---

## Жорсткий режим виконання (Strict Mode)

- Включаємо опції bash, які допомагають ловити помилки раніше:

```bash
set -euo pipefail
IFS=$'\n\t'
```

- Пояснення:

  - `-e` — вихід при першій помилці
  - `-u` — помилка при використанні необ’явлених змінних
  - `-o pipefail` — помилка у конвеєрі, якщо будь-яка команда не успішна
  - `IFS` — внутрішній роздільник полів, налаштований безпечніше

---

## Контроль змінних та конвеєрів

- Уникайте глобальних змінних без потреби.
- Використовуйте локальні змінні у функціях (`local var=value`).
- Перевіряйте статус виконання команд (`$?`).
- Для конвеєрів використовуйте `pipefail` (як вище), щоб не пропустити помилки.

---

## Оголошення конфігурації

- Виносьте налаштування (шляхи, змінні) у верхню частину скрипта або у зовнішній конфігураційний файл.
- Це полегшує зміну параметрів без редагування коду.

---

## Динамічні змінні (Дати)

- Для бекапів часто потрібні динамічні імена файлів з датою:

```bash
BACKUP_DIR="/backup"
DATE=$(date +'%Y-%m-%d_%H-%M-%S')
BACKUP_FILE="${BACKUP_DIR}/backup_${DATE}.tar.gz"
```

- Використовуйте форматування дати для зручності і унікальності імен.

---

## Функція `print_usage`
```bash
print_usage() {
  echo "Використання: $0 -s <source_dir> -d <backup_dir>"
}
```
- Виводить інструкцію як запускати скрипт.
- `$0` — ім’я скрипта.
- Чи підставляються значення для `-s <source_dir>` і `-d <backup_dir>` автоматично? Ці значення ти передаєш самому скрипту під час запуску з командного рядка
```bash
./backup.sh -s /home/user/documents -d /mnt/backup
```

---

## Функція `log`
```bash
log() {
  local level="$1"
  local msg="$2"
  echo "[$(date '+%Y-%m-%d %H:%M:%S')] [$level] $msg"
}
```
- Функція для логування повідомлень з позначенням часу і рівня логування (`INFO`, `ERROR` і т.д.).
- Приймає два аргументи: рівень логування і текст повідомлення.

---

## Ініціалізація змінних
```bash
SOURCE_DIR=""
BACKUP_DIR=""
```
- Змінні для зберігання шляху до директорії, яку треба бекапити, та директорії для збереження бекапу.

---

## Обробка параметрів командного рядка
Розглянемо цей фрагмент:
```bash
while getopts "s:d:h" opt; do
  case "$opt" in
    s) SOURCE_DIR="$OPTARG" ;;
    d) BACKUP_DIR="$OPTARG" ;;
    h) print_usage; exit 0 ;;
    *) print_usage; exit 1 ;;
  esac
done
```
- `getopts` обробляє опції `-s` (джерело), `-d` (директорія для бекапу), `-h` (допомога).
- Якщо вказано `-h`, виводиться допомога і скрипт завершується.
- Якщо вказано невідому опцію — виводиться допомога і вихід з помилкою.

## Звідки береться `$OPTARG`?
`$OPTARG` — це спеціальна змінна, яку встановлює команда `getopts` під час обробки параметрів.

- `getopts "s:d:h"` каже: шукай опції `-s`, `-d` (які очікують аргумент, бо після них стоїть двокрапка `:`) і `-h` (без аргументу).
- Коли скрипт запускають, наприклад:
  
  ```bash
  ./backup.sh -s /home/user/documents -d /mnt/backup
  ```

- `getopts` по черзі "збирає" опції і їх аргументи:
  - при зустрічі `-s /home/user/documents`:
    - `opt` буде `s`
    - `$OPTARG` буде `/home/user/documents`
  - при зустрічі `-d /mnt/backup`:
    - `opt` буде `d`
    - `$OPTARG` буде `/mnt/backup`

- Тобто `$OPTARG` — це значення, яке йде після опції, наприклад, після `-s` або `-d`.

## Перевірка обов’язкових параметрів
```bash
if [[ -z "$SOURCE_DIR" || -z "$BACKUP_DIR" ]]; then
  log "ERROR" "Не вказані обов’язкові параметри"
  print_usage
  exit 1
fi
```
- Якщо не передані обидві обов’язкові опції (`-s` і `-d`), виводиться помилка, допомога і скрипт завершується.

---

## Перевірка існування директорії джерела
```bash
if [[ ! -d "$SOURCE_DIR" ]]; then
  log "ERROR" "Джерело не існує: $SOURCE_DIR"
  exit 1
fi
```
- Якщо директорія джерела не існує, виводиться помилка і скрипт завершується.

---

## Створення директорії для бекапу (якщо не існує)
```bash
mkdir -p "$BACKUP_DIR"
```
- Команда `mkdir -p` створює директорію для бекапу, якщо її немає.

---

## Формування імені файлу бекапу з датою
```bash
DATE=$(date +'%Y-%m-%d_%H-%M-%S')
BACKUP_FILE="${BACKUP_DIR}/backup_${DATE}.tar.gz"
```
- Змінна `DATE` містить поточну дату і час у форматі `YYYY-MM-DD_HH-MM-SS`.
- Формується повний шлях до файлу бекапу з унікальним іменем.

---

## Логування початку процесу бекапу
```bash
log "INFO" "Початок бекапу $SOURCE_DIR у $BACKUP_FILE"
```
- Виводиться інформаційне повідомлення про початок бекапу.

---

## Створення архіву бекапу
```bash
tar czf "$BACKUP_FILE" -C "$SOURCE_DIR" .
```
- Команда `tar` створює стиснутий архів (`.tar.gz`).
- Опція `-C "$SOURCE_DIR"` змінює робочу директорію на `$SOURCE_DIR`.
- `.` означає архівувати всі файли і папки в цій директорії.

---

## Логування завершення процесу
```bash
log "INFO" "Бекап завершено успішно"
```
- Виводиться повідомлення про успішне завершення бекапу.

---

## Висновок

Цей скрипт приймає дві обов’язкові опції: 
   - директорію для бекапу 
   - директорію для збереження архіву. 

Він перевіряє правильність параметрів і існування директорії, створює архів з датою у назві, веде логування процесу і гарантує коректне завершення при помилках.

- `getopts` розбирає ці параметри.
- `$OPTARG` — це аргумент, який йде після опції (наприклад, шлях до директорії).
- Ці значення записуються у змінні `SOURCE_DIR` і `BACKUP_DIR` для подальшого використання у скрипті.
