# Загальні відомості про cron

## 1. Як працює демон cron

- **cron** — це системний демон (служба), який працює у фоновому режимі та періодично виконує заплановані завдання (cron jobs).
- Він зчитує конфігураційні файли crontab, у яких описані завдання та час їх запуску.
- Демон перевіряє кожну хвилину, чи потрібно запустити якесь завдання.
- Після запуску завдання виконується у окремому процесі.

### Певні нюани на що потрібно звертати увагу
- Cron запускає завдання у дуже обмеженому середовищі.
- Змінні середовища, які ви бачите у вашій консольній сесії (наприклад, `PATH`, `HOME`, `USER`), **не обов’язково встановлені** у середовищі cron.
- Через це команди можуть не знайти потрібні утиліти або скрипти.
- Наприклад, команда `python` може не знайтись, якщо `PATH` не містить `/usr/bin` або `/usr/local/bin`.

### Повні шляхи

- Завжди використовуйте **повні шляхи** до команд і файлів у скриптах, які запускає cron.
  
  Наприклад:

  ```bash
  /usr/bin/python3 /home/user/scripts/myscript.py
  ```

- Щоб дізнатись повний шлях до команди, використовуйте:

  ```bash
  which python3
  ```

- Це допомагає уникнути помилок, коли cron не знаходить команду через обмежений `PATH`.

## 2. Відмінності системного crontab від користувацького

- **crontab** — це скорочення від **cron table** (таблиця cron), файл або набір файлів, у яких описані правила запуску завдань cron.
- Кожен користувач системи може мати свій власний crontab, а також існує системний crontab.
- У crontab описується, **коли** і **які команди** потрібно виконувати.
- Формат файлу crontab — це текстовий файл, де кожен рядок — це окреме завдання з описом часу запуску та команди.

| Характеристика         | Системний crontab                  | Користувацький crontab                |
|-----------------------|----------------------------------|-------------------------------------|
| Розташування          | `/etc/crontab`, `/etc/cron.d/*`  | Кожен користувач має свій файл, який можна редагувати через `crontab -e` |
| Формат                | Має додаткове поле для користувача (user), який запускає завдання | Не містить поля user — завдання виконується від імені користувача |
| Права редагування     | Потрібні права root               | Користувач може редагувати лише свій crontab |
| Призначення           | Для системних завдань, що запускаються від імені різних користувачів | Для персональних завдань користувача |

## 3. Синтаксис п’яти полів часу у crontab
У crontab для позначення часу використовується п’ять полів (хвилина, година, день місяця, місяць, день тижня), і в кожному полі можна використовувати спеціальні символи. Найпопулярніший — це **зірочка (*)**.

- **Зірочка (*) означає "кожне можливе значення" для цього поля.**

Формат запису часу у crontab:

```
* * * * *
| | | | |
| | | | +---- День тижня (0-7) (0 або 7 — неділя)
| | | +------ Місяць (1-12)
| | +-------- День місяця (1-31)
| +---------- Година (0-23)
+------------ Хвилина (0-59)
```

### Приклади:

- `* * * * *` — кожну хвилину
- `0 * * * *` — на початку кожної години
- `0 0 * * *` — щоденно опівночі
- `0 0 1 * *` — щомісячно першого числа опівночі
- `0 0 1 1 *` — щорічно першого січня опівночі

## 4. Методи керування завданнями через crontab

- **Перегляд crontab користувача:**

  ```bash
  crontab -l
  ```

- **Редагування crontab:**

  ```bash
  crontab -e
  ```

- **Видалення crontab:**

  ```bash
  crontab -r
  ```

- **Перегляд crontab іншого користувача (потрібні права root):**

  ```bash
  crontab -u username -l
  ```

---

## 5. Передача змінних середовища та логування

- За замовчуванням cron має мінімальний набір змінних середовища.
- Щоб передати змінні, можна оголосити їх безпосередньо у crontab:

  ```
  PATH=/usr/local/bin:/usr/bin:/bin
  SHELL=/bin/bash
  ```

- Також можна експортувати змінні в самому скрипті.

- Для логування стандартного виводу та помилок у файл додають перенаправлення:

  ```
  * * * * * /path/to/script.sh >> /var/log/script.log 2>&1
  ```

  - `>>` — додавання в кінець файлу (щоб не перезаписувати)
  - `2>&1` — перенаправлення stderr у stdout

## 6. Алгоритми діагностики помилок

- **Перевірка логів:**

  - За замовчуванням вивід cron може надсилатися на email користувача.
  - Перевіряйте файли логів, наприклад `/var/log/cron` або `/var/log/syslog` (залежить від системи).

- **Тестування скриптів вручну:**

  - Запускайте скрипти поза cron, щоб впевнитися, що вони працюють.

- **Перевірка прав доступу:**

  - Файли та скрипти повинні мати права на виконання.
  - Шляхи до команд і файлів мають бути абсолютними.

- **Використання повних шляхів у скриптах і crontab**

- **Додавання виводу в лог для відлагодження**

- **Перевірка змінних середовища**

- **Перевірка синтаксису crontab**

  ```bash
  crontab -l | grep -v '^#' | while read line; do echo "$line"; done
  ```
## Різниця між cron та crontab

| Поняття              | Опис                                                                 |
|----------------------|----------------------------------------------------------------------|
| **cron**             | Демон (служба), який запускає заплановані завдання у фоновому режимі. |
| **crontab**           | Файл (або набір файлів), який містить правила (розклад + команди) для cron. |

- Тобто **cron** — це програма, що виконує завдання іншими словами це "двигун", який працює постійно.
- **crontab** — це конфігураційний файл, який задає, що і коли виконувати або "план задач", який cron регулярно читає і виконує.

## Як Додавати задачі для cron вручну?

### 1. Редагування системного файлу `/etc/crontab`

- Файл `/etc/crontab` — це **системний crontab**, який відрізняється від користувацьких тим, що в ньому є додаткове поле — ім’я користувача, від імені якого виконувати команду:

  Формат рядка в `/etc/crontab`:

  ```
  * * * * * user command
  ```

  Наприклад:

  ```
  0 2 * * * root /usr/local/bin/backup.sh
  ```

- Цей файл можна редагувати вручну будь-яким текстовим редактором (`vi`, `nano` і т.д.).

- Після збереження cron автоматично перечитає цей файл і почне виконувати завдання за розкладом.

---

### 2. Файли у директорії `/etc/cron.d/`

- У цій директорії можна створювати власні файли з розкладом завдань у тому ж форматі, що і в `/etc/crontab` (з полем користувача).

- Наприклад, створити файл `/etc/cron.d/myjob` зі вмістом:

  ```
  30 1 * * * root /usr/local/bin/daily_task.sh
  ```

- cron автоматично підхоплює всі файли з цієї директорії.

---

### 3. Файли у `/etc/cron.hourly/`, `/etc/cron.daily/`, `/etc/cron.weekly/`, `/etc/cron.monthly/`

- Якщо додати скрипт у одну з цих директорій, він буде виконуватися відповідно щогодини, щодня, щотижня або щомісяця.

- Просто скопіюйте свій скрипт туди і переконайтеся, що він має права на виконання (`chmod +x`).

---

### 4. Редагування користувацького crontab вручну (не рекомендується)

- Можна було (і можна) вручну редагувати файл користувацького crontab, який знаходиться у:

  ```
  /var/spool/cron/crontabs/username
  ```

- Але **це не рекомендується**, бо:

  - cron може не помітити зміни, якщо файл не оновлюється правильно.
  - Потрібно дотримуватись правильного формату.
  - Краще використовувати команду `crontab -e`, яка автоматично виконує перевірку і повідомляє про помилки.

---

## Чому все працювало?

- Якщо редагували **системний файл `/etc/crontab`** або файли у `/etc/cron.d/`, то cron автоматично перечитує ці файли і виконує завдання.

- Якщо додавали скрипти у спеціальні директорії `/etc/cron.daily/` то вони виконувалися автоматично.

- Якщо вручну редагували файли у `/var/spool/cron/crontabs/`, можливо, це працювало, але це не найкраща практика.

## Кроки для додавання власного завдання

### Крок 1. Створіть скрипт

Наприклад, створимо простий скрипт, який записує дату у лог:

```bash
sudo nano /etc/cron.daily/my_custom_task
```

Вміст файлу:

```bash
#!/bin/bash
echo "Custom daily task run at $(date)" >> /var/log/my_custom_task.log
```

---

### Крок 2. Зробіть скрипт виконуваним

```bash
sudo chmod +x /etc/cron.daily/my_custom_task
```

---

### Крок 3. Перевірте права і власника

- Зазвичай скрипти у цих директоріях належать користувачу `root`.
- Переконайтеся, що власником є `root`:

```bash
sudo chown root:root /etc/cron.daily/my_custom_task
```

---

### Крок 4. Перевірте роботу

- Скрипти у цих директоріях запускаються за розкладом (щодня, щотижня і т.д.) згідно з налаштуваннями у `/etc/crontab`.
- Для тесту можна запустити скрипт вручну:

```bash
sudo /etc/cron.daily/my_custom_task
```

- Перевірте, чи створився лог:

```bash
cat /var/log/my_custom_task.log
```

## Підсумок

| Спосіб додавання задачі       | Де редагувати                | Особливості                              | Рекомендації                      |
|------------------------------|-----------------------------|-----------------------------------------|----------------------------------|
| Користувацький crontab        | `crontab -e` (файл у `/var/spool/cron/crontabs/username`) | Безпечно, автоматично перевіряється | Найкращий спосіб для користувача |
| Системний crontab             | `/etc/crontab`               | Має поле користувача, можна редагувати вручну | Добре для системних задач        |
| Файли у `/etc/cron.d/`        | Створення файлів у цій папці | Аналогічний формат як у `/etc/crontab` | Зручно для пакетів та адміністрування |
| Скрипти у `/etc/cron.daily/` і т.д. | Поміщення скриптів у відповідні папки | Виконуються періодично автоматично       | Зручно для регулярних завдань    |

---

Якщо ви бачили, що на сервері "додавали задачі вручну", найімовірніше це було:

- Редагування **системного `/etc/crontab`** або
- Додавання файлів у `/etc/cron.d/` або
- Поміщення скриптів у `/etc/cron.daily/` тощо.


## Важливо

- Не ставте довгі завдання у ці скрипти, які можуть довго виконуватися, оскільки це може блокувати виконання інших скриптів у тій же директорії.
- Якщо потрібно тонке налаштування часу, краще додавати завдання у власний crontab через `crontab -e` або у `/etc/cron.d/`.
