
# Віддалене підключення та деплой застосунку

## 1. Робочий доступ через SSH

- **Приклад підключення:**

```bash
ssh -i ~/.ssh/test_goit -p 2211 goit@127.0.0.1
```

- Тут:
  - `-i ~/.ssh/test_goit` — приватний ключ для аутентифікації.
  - `-p 2211` — нестандартний порт.
  - `goit@127.0.0.1` — користувач `goit` на локальній машині (можна замінити на IP/домен сервера).

- **SSH — це основний інструмент для:**
  - Адміністрування серверів.
  - Доставки коду.
  - Автоматизації завдань.

---

## 2. SSH як щоденний інструмент

- **Типовий робочий цикл:**

  - Підключення до сервера.
  - Перевірка стану застосунку.
  - Оновлення коду.
  - Перезапуск сервісів.
  - Перевірка логів.

- **Важливо:** Будь-яка команда може вплинути на продакшн, тому дії мають бути передбачуваними та безпечними.

---

## 3. Контекст користувача

- Підключайтесь під користувачем з мінімальними правами (не root).
- Для дій, що потребують прав адміністратора, використовуйте `sudo`.
- Це знижує ризики випадкових помилок.

---

## 4. Підключення у сценаріях автоматизації

- Приклад автоматичного деплою через SSH у скрипті:

```bash
ssh -i ~/.ssh/test_goit -p 2211 goit@127.0.0.1 << EOF
  cd /opt/myapp
  git pull origin main
  ./restart.sh
EOF
```

- Використання ключів дозволяє запускати безпарольні підключення.

---

## 5. Організація простору для застосунку

- **Стандартне місце для сторонніх застосунків — `/opt`**

  - `/opt` — призначена для додаткового програмного забезпечення, яке не є частиною базової системи.
  - Зручно підтримувати, оновлювати і не заважати системним файлам.

- **Приклад створення структури:**

```bash
sudo mkdir -p /opt/myapp
sudo chown goit:goit /opt/myapp
```

- Працюємо під користувачем `goit`, не під root.

---

## 6. Пастка хаотичного розміщення

- Хаотичне розміщення файлів у домашніх папках користувачів або в `/usr` ускладнює підтримку.
- `/opt` — це чистий простір для вашого застосунку.

---

## 7. Практика: створення ієрархії

```bash
sudo mkdir -p /opt/myapp/releases
sudo mkdir -p /opt/myapp/shared
sudo chown -R goit:goit /opt/myapp
```

- `releases` — для версій коду.
- `shared` — для конфігів, логів, даних.

---

## 8. Доставка коду

### SCP — для простих разових копіювань

```bash
scp -i ~/.ssh/test_goit -P 2211 app.tar.gz goit@127.0.0.1:/opt/myapp/releases/
```

- Недоліки:
  - Повне копіювання.
  - Не враховує зміни.
  - Не підходить для частих оновлень.

---

### Rsync — індустріальний стандарт

```bash
rsync -avz -e "ssh -i ~/.ssh/test_goit -p 2211" ./app/ goit@127.0.0.1:/opt/myapp/releases/current/
```

- Переваги:
  - Передає лише змінені файли.
  - Підтримує видалення зайвих файлів (`--delete`).
  - Швидко і ефективно.

---

## 9. Відокремлення коду від конфігурації

- Використовуємо `.env` файли для зберігання налаштувань:

```
DATABASE_URL=postgres://user:pass@host:5432/dbname
SECRET_KEY=supersecretvalue
DEBUG=False
```

- **Практика:**
  - `.env` не зберігаємо в репозиторії.
  - Конфігурація зберігається окремо або в секретах хмарних провайдерів (AWS Secrets Manager, GCP Secret Manager).
  - При деплої підвантажуємо `.env` або налаштовуємо змінні оточення.

---

## 10. Перевірка після доставки

- Перевірити, що файли на сервері:

```bash
ssh -i ~/.ssh/test_goit -p 2211 goit@127.0.0.1 "ls -l /opt/myapp/releases/current"
```

- Перевірити цілісність:

  - Хеш-сумми.
  - Логи запуску.

---

## 11. Ізоляція та середовище

### Чому середовище має бути ізольованим?

- Щоб уникнути конфліктів між залежностями різних проектів.
- Щоб не впливати на глобальні бібліотеки системи.
- Для стабільності і передбачуваності.

---

### Створення та активація virtualenv

```bash
python3 -m venv /opt/myapp/venv
source /opt/myapp/venv/bin/activate
pip install -r requirements.txt
```

- Можна автоматизувати у скрипті:

```bash
#!/bin/bash
cd /opt/myapp/releases/current
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

---

### Встановлення залежностей

- `requirements.txt` — список залежностей.
- При оновленні коду — оновлюємо залежності.

---

### Конфігурація через .env

- Приклад для AWS:

```bash
aws secretsmanager get-secret-value --secret-id myapp/env --query SecretString --output text > /opt/myapp/shared/.env
```

- Підвантажуємо `.env` у середовище перед запуском.

---

### Ручний запуск — обов'язковий крок

- Після доставки і налаштування середовища запускаємо застосунок:

```bash
source /opt/myapp/venv/bin/activate
python app.py
```

- Перевіряємо, що все працює.

---

## 12. Перетворення застосунку на systemd-сервіс

- Приклад файлу `/etc/systemd/system/myapp.service`:

```ini
[Unit]
Description=MyApp Service
After=network.target

[Service]
User=goit
WorkingDirectory=/opt/myapp/releases/current
ExecStart=/opt/myapp/venv/bin/python app.py
Restart=always

[Install]
WantedBy=multi-user.target
```

- Команди:

```bash
sudo systemctl daemon-reload
sudo systemctl enable myapp
sudo systemctl start myapp
sudo systemctl status myapp
```

- Переваги:
  - Автоматичний запуск при завантаженні системи.
  - Автоматичне відновлення після збоїв.

---

# Підсумок

- SSH — основний інструмент деплою і адміністрування.
- Використання `/opt` як стандартного каталогу для застосунків.
- Rsync — ефективний спосіб доставки коду.
- Відокремлення конфігурації через `.env`.
- Ізоляція середовища через virtualenv.
- Запуск через systemd для стабільності.
