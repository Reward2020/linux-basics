
# Основи мережі в Linux для розробників: практичний погляд

Розуміння мережі — це не лише про налаштування серверів чи адміністрування, а й про те, як ваш код взаємодіє з мережею, як перевірити доступність сервісів, діагностувати проблеми і не потрапити у типові пастки. Тому почнемо з інструментів, які допоможуть вам швидко орієнтуватися в мережевій ситуації вашого Linux-середовища.

## Інвентаризація мережі: що є у вашій системі?

Перш ніж щось налаштовувати або діагностувати, потрібно розуміти, які інтерфейси у вас є, їхній стан і IP-адреси.

Команда `ip` — сучасний інструмент замість старого `ifconfig`. Вона допомагає подивитись на мережеві інтерфейси, їхній стан і налаштування.

- `ip link` показує список мережевих інтерфейсів і їхній стан («труби»):

```bash
ip link
```

Вивід покаже, які інтерфейси активні, їхні імена (наприклад, `eth0`, `lo`), MAC-адреси, статус (UP/DOWN).

- `ip addr` — паспорт інтерфейсу, тут ви бачите IP-адреси (IPv4 та IPv6), призначені інтерфейсу:

```bash
ip addr
```

Важливо зрозуміти, що таке loopback (`lo`) — це «внутрішнє дзеркало» системи, IP `127.0.0.1` — це адреса для звернення до самого себе. Вона не виходить у мережу, але часто використовується для тестування локальних сервісів.

Також варто знати різницю між `IPv4` (наприклад, `192.168.1.10`) та `IPv6` (наприклад, `fe80::1ff:fe23:4567:890a`). Сучасні системи підтримують обидва протоколи, і іноді це може впливати на поведінку додатків.


### **Для публічного IP через термінал**
- Введіть:
```bash
curl ifconfig.me
```
або
```bash
curl ipinfo.io/ip
```
  Це покаже ваш зовнішній (публічний) IP-адрес [1].

## Перевірка зв’язності: ping як ехолокація

Коли ваш сервіс не відповідає, перше, що робить розробник — це пінгує адресу, щоб перевірити, чи до неї взагалі можна дістатися.

Команда `ping` відправляє ICMP пакети і чекає відповіді — це базова перевірка мережевої доступності.

```bash
ping -c 4 8.8.8.8
```

Вивід покаже час відповіді (latency) і чи взагалі є зв’язок.

Якщо ви пінгуєте домен, наприклад:

```bash
ping -c 4 google.com
```

це одночасно перевіряє роботу DNS — чи може система розв’язати ім’я в IP.

Якщо ping мовчить (немає відповіді), це може означати:

- Проблеми з мережею (відсутність маршруту)
- Фаєрвол блокує ICMP
- Цільовий хост не працює

Щоб зрозуміти, де «зависла» мережа, можна використовувати `traceroute` — він показує шлях пакетів від вас до цілі:

```bash
traceroute google.com
```
```bash
sudo traceroute -T google.com
sudo traceroute -4 google.com
```

Це допомагає знаходити «вузькі місця» у мережі.

## Аудит відкритих портів та сокетів: netstat vs ss

Розробникам часто потрібно знати, які порти зайняті, які сервіси слухають на яких адресах, щоб уникнути конфліктів.

Раніше для цього використовували `netstat`, але зараз рекомендують `ss` — він швидший і точніший.

Щоб побачити всі відкриті TCP та UDP порти, які слухають на системі:

```bash
ss -tuln
```

- `-t` — TCP
- `-u` — UDP
- `-l` — listening (слухаючі)
- `-n` — показувати порти у числовому вигляді (без резолвінгу)

Вивід покаже локальні адреси і порти, а також PID процесів, які їх використовують (якщо додати `-p`).

Це допомагає знайти, чи не зайнятий потрібний порт іншим процесом.

### Перевірте, чи немає локальних правил фаєрвола (iptables, ufw), які блокують ICMP:
```bash
sudo iptables -L -v -n
sudo ufw status
```

## Binding: як сервіси «прив’язуються» до інтерфейсів?

Binding — це прив’язка сервісу до конкретної IP-адреси або інтерфейсу.

- Якщо сервіс слухає на `127.0.0.1` (localhost), він доступний лише локально (з тієї ж машини).
- Якщо на `0.0.0.0`, то він слухає на всіх інтерфейсах — відкритий для зовнішніх підключень.
- Можна прив’язати і до конкретної IP-адреси інтерфейсу.

Це важливо, бо неправильний binding може призвести до того, що сервіс працює, але недоступний зовні.

---

### Приклад у Python (binding на localhost)

```python
import socket

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM))
    s.bind(('127.0.0.1', 8081))
    s.listen(1)
    print("Сервер запущено на 127.0.0.1:8081. Чекаю на підключення...")while True:
       conn, addr = s.accept()
       print(f"З'єднання від: {addr}")
       conn.sendall(b"Hello from Server!")
       conn.close()
```

Цей сервер буде доступний лише з тієї ж машини.
### Запуск та перевірка:
   * Термінал 1: python3 server.py
      * Термінал 2: curl 127.0.0.1:8081 (Працює!)
      * Термінал 2: Спробуй підключитися через зовнішній IP машини: curl <IP_ВАШОЇ_VM>:8081 (Помилка! Connection refused).
   3. Виконай в системі:
   `ss -tulpn | grep 8081`. Побачите 127.0.0.1:8081. що це "внутрішні двері".
   4. Виправлення: Зміни в коді 127.0.0.1 на 0.0.0.0, перезапусти і  тепер `ss` каже `*:8081`, і сервер доступний ззовні.
---

```python
import socket
# Створюємо TCP сокет
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
# Біндимо на зовні 
   s.bind(('0.0.0.0', 8081))
   s.listen(1)
   print("Сервер запущено на 127.0.0.1:8081. Чекаю на підключення...")while True:
       conn, addr = s.accept()
       print(f"З'єднання від: {addr}")
       conn.sendall(b"Hello from Server!")
       conn.close()
```

Тут сервер доступний із зовні.

Правильний інструмент. Оскільки ми тестуємо TCP-з'єднання, а не Веб-сайт, замість `curl` використай утиліту telnet або nc (netcat).
```bash
nc 127.0.0.1 8081
```
"Кричати" на curl (швидкий фікс). Якщо хочеш змусити `curl` прийняти твій сирий текст, додай прапорець --http0.9:
```bash
curl 127.0.0.1:8081 --http0.9
```

## Практичні сценарії: діагностика проблем з доступністю сервісу

Уявімо, що ваш сервіс запущений, але клієнти його не бачать.
1. Запускаємо "прослушку" (Термінал 1):
```
sudo tcpdump -i lo port 8081 -S
```
2. Робимо запит (Термінал 2):
```
curl 127.0.0.1:8081
```
3. Аналіз: Прапорці в терміналі з tcpdump:

| Прапорець | Значення | Що відбувається |
| :-- | :-- | :-- |
| [S] | SYN | Клієнт починає з’єднання (привіт) |
| [S.] | SYN + ACK | Сервер відповідає (привіт, отримав) |
| [.] | ACK | Клієнт підтверджує (починаємо) |
| [P] | PUSH + ACK | Дані передаються (HTTP GET) |
| [F] | FIN | означає, що сторона, яка відправляє цей пакет, хоче закрити TCP-з’єднання. |




### Конфлікт портів та пошук "винуватця"

   1. Симулюємо проблему:
   Залиш server.py запущеним. Спробуй запустити його ще раз у другому вікні.
   * Отримаємо: OSError: [Errno 98] Address already in use.
   2. Шукаємо процес:
   ```
   sudo lsof -i :8081
   ```
   (Або sudo netstat -tulpn | grep 8081).
   3. Вбиваємо заважаючий процес:
   ```
   sudo kill -9 <PID_З_ПОПЕРЕДНЬОЇ_КОМАНДИ>
   ```

1. **Перевірка власної адреси**  
   Переконайтесь, що інтерфейс активний і має IP:

   ```bash
   ip addr show
   ```

2. **Перевірка зв’язності**  
   Пінгуйте адресу сервера з клієнта (або навпаки), щоб перевірити маршрут.

3. **Перевірка порту і binding**  
   На сервері подивіться, чи сервіс слухає потрібний порт і на правильній адресі:

   ```bash
   ss -tuln | grep :PORT
   ```

   Якщо сервіс прив’язаний до `127.0.0.1`, а клієнт зовнішній — підключення не буде.

4. **Перевірка через curl**  
   Якщо це HTTP-сервіс, спробуйте звернутися локально і зовні:

   ```bash
   curl http://localhost:PORT
   curl http://SERVER_IP:PORT
   ```

   Це допоможе відрізнити проблему binding від проблеми мережі.

## Мережа в Docker (Місток у DevOps)

1. Запускаємо контейнер:
```bash
sudo docker run -d --name web-test -p 8081:80 nginx
``` 
2. Демонструємо мапінг портів:
Виконай `ss -tulpn | grep 8081`. Покажи, що процес docker-proxy слухає порт на хостовій машині.
3. Заходимо "всередину" мережі:
```bash
sudo docker inspect web-test | grep IPAddress
```
Спробуй зробити curl <IP_КОНТЕЙНЕРА>:80 прямо з хоста. Це покаже студентам, як працює Bridge-мережа Docker.

Коли ти запускаєш контейнер з прапорцем -d (detached mode), він працює у фоновому режимі. Щоб його зупинити та видалити, використовуй наступні команди:
###  Зупинка контейнера
Це надішле процесу всередині сигнал SIGTERM, щоб він коректно завершив роботу:
```bash
docker stop web-test
```

###  Видалення контейнера
Після зупинки контейнер все ще займає місце в системі (його можна побачити через docker ps -a). Щоб видалити його повністю:
```bash
sudo docker rm web-test
```

* Все в одній команді: Якщо хочеш зупинити і одразу видалити:
```bash
sudo docker rm -f web-test
```

* Очищення після себе: `--rm`. Якщо запустити так:
`docker run -d --rm --name web-test -p 8081:80 nginx`, то після виконання docker stop web-test контейнер автоматично видалиться сам. Це золотий стандарт для тимчасових тестів.
* Як перевірити, чи він ще є:
```bash
docker ps       # Тільки запущені
docker ps -a    # Всі, включаючи зупинені
```

## Підсумок

Знання команд `ip`, `ping`, `ss` — базовий мінімум для розробника, щоб швидко діагностувати мережеві проблеми. Розуміння binding допоможе уникнути ситуацій, коли сервіс працює, але недоступний зовні. А `traceroute` — інструмент для глибшого аналізу мережевого шляху.




## Де зберігаються налаштування мережі в Linux?

В залежності від дистрибутива Linux, налаштування мережі можуть зберігатися у різних файлах та керуватися різними системами. Ось основні варіанти:

### 1. Debian/Ubuntu (NetworkManager, systemd-networkd, /etc/network/interfaces)

- **NetworkManager** — сучасний менеджер мережі, використовується у більшості десктопних дистрибутивів.
- **systemd-networkd** — часто використовується на серверах.
- **/etc/network/interfaces** — класичний спосіб для старих систем або серверів без NetworkManager.

#### Файл /etc/network/interfaces

Тут можна вручну прописати IP-адресу, шлюз, DNS:

```bash
sudo nano /etc/network/interfaces
```

Приклад статичної конфігурації:
```
auto eth0
iface eth0 inet static
    address 192.168.1.100
    netmask 255.255.255.0
    gateway 192.168.1.1
    dns-nameservers 8.8.8.8 8.8.4.4
```

Після змін — перезапустіть мережу:
```bash
sudo systemctl restart networking
```

#### NetworkManager (GUI/CLI)

- GUI: nm-connection-editor
- CLI: nmcli

Приклад зміни IP через nmcli:
```bash
nmcli con show
nmcli con mod "Wired connection 1" ipv4.addresses 192.168.1.100/24
nmcli con mod "Wired connection 1" ipv4.gateway 192.168.1.1
nmcli con mod "Wired connection 1" ipv4.dns 8.8.8.8
nmcli con mod "Wired connection 1" ipv4.method manual
nmcli con up "Wired connection 1"
```

#### systemd-networkd

Файли конфігурації лежать у /etc/systemd/network/
Приклад:
```bash
sudo nano /etc/systemd/network/20-wired.network
```
```
[Match]
Name=eth0

[Network]
Address=192.168.1.100/24
Gateway=192.168.1.1
DNS=8.8.8.8
```
Після змін:
```bash
sudo systemctl restart systemd-networkd
```

### 2. RedHat/CentOS/Fedora (ifcfg, NetworkManager)

- **/etc/sysconfig/network-scripts/ifcfg-eth0** — класичний спосіб.
- **NetworkManager** — сучасний спосіб (аналогічно Ubuntu).

#### Файл ifcfg-eth0

```bash
sudo nano /etc/sysconfig/network-scripts/ifcfg-eth0
```
```
DEVICE=eth0
BOOTPROTO=static
ONBOOT=yes
IPADDR=192.168.1.100
NETMASK=255.255.255.0
GATEWAY=192.168.1.1
DNS1=8.8.8.8
```
Після змін:
```bash
sudo systemctl restart network
```

### 3. DNS-налаштування

- /etc/resolv.conf — тут прописуються DNS-сервери.
- У сучасних системах цей файл може генеруватися автоматично (наприклад, systemd-resolved).

### 4. Тимчасова зміна IP (до перезавантаження)

```bash
sudo ip addr add 192.168.1.100/24 dev eth0
sudo ip route add default via 192.168.1.1
```
Але після перезавантаження ці зміни зникнуть.

### Як Ubuntu 22.04 керує мережею "з коробки"?

1. **NetworkManager** — використовується за замовчуванням на десктопах.
2. **netplan** — основний спосіб опису мережевої конфігурації у сучасних Ubuntu (з 17.10+).
3. **systemd-networkd** — може використовуватись на серверах, але зазвичай через netplan.

---

## Де шукати налаштування мережі в Ubuntu 22.04?

### 1. Netplan

- Основні файли:  
  /etc/netplan/*.yaml

Зазвичай там є файл на кшталт 01-netcfg.yaml, 50-cloud-init.yaml або подібний.

#### Приклад вмісту файлу netplan:

```yaml
network:
  version: 2
  renderer: NetworkManager
  ethernets:
    enp3s0:
      dhcp4: true
```

- **renderer: NetworkManager** — означає, що фактичне керування інтерфейсом здійснює NetworkManager (типово для десктопів).
- **renderer: networkd** — для серверів.

#### Як змінити налаштування?

1. Відкрийте файл:
   ```bash
   sudo nano /etc/netplan/01-network-manager-all.yaml
   ```
   (або інший файл у цій папці)

2. Внесіть зміни (наприклад, для статичної IP-адреси):

   ```yaml
   network:
     version: 2
     renderer: NetworkManager
     ethernets:
       enp3s0:
         dhcp4: no
         addresses: [192.168.1.100/24]
         gateway4: 192.168.1.1
         nameservers:
           addresses: [8.8.8.8,8.8.4.4]
   ```

3. Застосуйте зміни:
   ```bash
   sudo netplan apply
   ```

---

### 2. NetworkManager

- NetworkManager зчитує налаштування з netplan, але також має свої профілі:
  - /etc/NetworkManager/system-connections/*.nmconnection

Ви можете керувати ними через GUI (nm-connection-editor) або через команду nmcli.

#### Переглянути всі підключення:
```bash
nmcli con show
```

#### Змінити IP через nmcli:
```bash
nmcli con mod "Wired connection 1" ipv4.addresses 192.168.1.100/24
nmcli con mod "Wired connection 1" ipv4.gateway 192.168.1.1
nmcli con mod "Wired connection 1" ipv4.dns 8.8.8.8
nmcli con mod "Wired connection 1" ipv4.method manual
nmcli con up "Wired connection 1"
```

## Висновок

- У Ubuntu 22.04 мережа керується через netplan (файли /etc/netplan/*.yaml).
- Якщо renderer: NetworkManager — керування йде через NetworkManager, але netplan все одно є "точкою входу".
- Старі файли (/etc/network/interfaces) не використовуються за замовчуванням.
- Для постійних змін — редагуйте відповідні конфігураційні файли.
- Для швидких тестів — використовуйте ip або ifconfig.
- Для сучасних систем — краще користуватися NetworkManager або systemd-networkd.
