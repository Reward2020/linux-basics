

## Практична демонстрація job control у Bash

### Цілі:
- Запустити процес у фоні (`&`)
- Побачити список job'ів (`jobs`)
- Поставити процес на паузу (`Ctrl+Z`)
- Відновити у фоні (`bg`)
- Повернути у передній план (`fg`)
- Запустити процес через `nohup` і вийти з сесії

---

### Крок 1: Створимо простий скрипт для демонстрації

Назвемо файл `demo_sleep.sh`:

```bash
nohup ./demo_sleep.sh > output.log 2>&1 &
```
```bash
cat << 'EOF' > demo_sleep.sh
#!/bin/bash
echo "Start script. PID $$"
for i in {1..10}; do
  echo "Step $i"
  sleep 3
done
echo "finish script"
EOF
```

- Скрипт виводить повідомлення кожні 3 секунди 10 разів.
- PID виводиться для наочності.

---

### Крок 2: Дамо права на виконання

```bash
chmod +x demo_sleep.sh
```

---

### Крок 3: Запуск у фоні

```bash
./demo_sleep.sh &
```

- Bash виведе номер job та PID, наприклад: `[1] 12345`
- Термінал залишається вільним для введення команд.

---

### Крок 4: Перегляд job'ів

```bash
jobs
```

Приклад виводу:

```
[1]+  Running                 ./demo_sleep.sh &
```

---

### Крок 5: Поставити процес на паузу

- Активуємо процес у передньому плані (якщо він у фоні):

```bash
fg %1
```

- Натискаємо `Ctrl+Z` — процес призупиниться, Bash виведе:

```
[1]+  Stopped                 ./demo_sleep.sh
```

---

### Крок 6: Відновити процес у фоні

```bash
bg %1
```

- Процес продовжить роботу у фоні.

---

### Крок 7: Повернути процес у передній план

```bash
fg %1
```

---

### Крок 8: Запуск процесу з `nohup` (щоб він не завершився при виході)

```bash
nohup ./demo_sleep.sh > demo_sleep.log 2>&1 &
```

- Процес буде працювати навіть після виходу з сесії SSH.
- Вивід перенаправлений у файл `demo_sleep.log`.

---

### Крок 9: Вийти з сесії (SSH) і перевірити, що процес продовжує працювати

```bash
exit
```

Після повторного підключення можна перевірити процес:

```bash
ps aux | grep demo_sleep.sh
```

---

## Підсумок

Цей простий скрипт і набір команд ілюструють основні інструменти job control у Bash:

- `&` — запуск у фоні
- `jobs` — перегляд job'ів
- `fg`, `bg` — керування процесом між переднім та фоновим планом
- `Ctrl+Z` — призупинка процесу
- `nohup` — запуск процесу, який не завершується при виході з терміналу

# Розширене пояснення Ctrl+Z і SIGTSTP у Bash

---

## Що важливо знати про `Ctrl+Z`

- `Ctrl+Z` надсилає процесу сигнал **SIGTSTP** — **Terminal Stop**.
- Процес **тимчасово призупиняється** (не виконується, не споживає CPU).
- Bash позначає його статус як **Stopped**.
- Термінал звільняється для введення нових команд.
- Процес не переходить у фон, а **заморожується**.

---

## Відмінність `Ctrl+Z` від запуску з `&`

| Особливість | `Ctrl+Z` | `&` (фон) |
|-------------|----------|-----------|
| Коли застосовується | До вже запущеного процесу | Під час запуску команди |
| Результат | Процес призупинений (Stopped) | Процес працює у фоні (Running) |
| Використання CPU | Ні | Так |
| Продовження роботи | Потрібно вручну відновити (`bg` або `fg`) | Працює одразу |
| Звільнення терміналу | Так | Так |

---

## Приклади використання Ctrl+Z

```bash
nano config.txt
# Натискаємо Ctrl+Z
# Отримаємо:
[1]+  Stopped                 nano config.txt

jobs
# Виведе:
[1]+  Stopped                 nano config.txt

bg %1
# Відновить nano у фоні (хоча для nano це не дуже зручно)

fg %1
# Поверне nano у передній план
```

---

## Практичні поради

- Якщо випадково запустив довгу команду у передньому плані і хочеш швидко звільнити термінал — натисни `Ctrl+Z`.
- Потім можеш або:
  - відновити процес у фоні (`bg`),  
  - або повернути у передній план (`fg`).
- Якщо хочеш, щоб процес працював після виходу з терміналу — запускай його з `nohup` або через `screen`/`tmux`.

---

## Короткий чеклист для роботи з `Ctrl+Z`

| Дія | Команда/Дія |
|------|-------------|
| Призупинити процес | `Ctrl+Z` |
| Переглянути призупинені та фоні процеси | `jobs` |
| Відновити процес у фоні | `bg %<номер job>` |
| Повернути процес у передній план | `fg %<номер job>` |
| Завершити процес | `kill %<номер job>` або `kill -9 %<номер job>` |

# Робота з job control у Bash: `Ctrl+Z`, `bg`, `fg`

---

## 1. `Ctrl+Z` — призупинення процесу

- Натискання `Ctrl+Z` надсилає процесу сигнал **SIGTSTP** (Terminal Stop).
- Процес переходить у стан **Stopped** (зупинений, «заморожений»).
- Він не виконується і не споживає CPU.
- Bash звільняє термінал для введення нових команд.
- Приклад виводу:
  ```
  [1]+  Stopped   nano config.php
  ```

---

## 2. Команда `bg` — відновлення процесу у фоні

- `bg` надсилає процесу сигнал **SIGCONT** (continue).
- Процес переходить зі статусу **Stopped** у **Running**.
- Запускає процес у фоні, не блокуючи термінал.
- Повертає контроль терміналу користувачу.
- Синтаксис:
  - `bg` — відновити останній зупинений процес.
  - `bg %<номер>` — відновити конкретний job.
- Приклад:
  ```bash
  bg %1
  ```
- Результат:
  ```
  [1]+ nano config.php &
  ```

---

## 3. Команда `fg` — повернення процесу у передній план

- Повертає зупинений або фоновий процес у передній план.
- Дозволяє взаємодіяти з програмою, бачити інтерфейс, зупиняти Ctrl+C.
- Синтаксис:
  - `fg` — повернути останній job.
  - `fg %<номер>` — повернути конкретний job.

---

## 4. Приклади типових сценаріїв

### PHP локальний сервер

```bash
php -S localhost:8000
# Помічаємо помилку, треба редагувати код
Ctrl+Z
bg
nano index.php
```

- Сервер продовжує працювати у фоні.
- Ти редагуєш код у редакторі.

---

### Node.js сервер

```bash
node app.js
Ctrl+Z
bg
npm install axios
```

---

### Python Flask

```bash
python3 app.py
Ctrl+Z
bg
htop
```

---

## 5. Перегляд та керування job’ами

- `jobs` — показує список зупинених та фонових процесів:

```
[1]-  Stopped   nano index.php
[2]+  Stopped   python3 app.py
```

- Відновлення конкретного процесу:

```bash
bg %2
fg %1
```

---

## 6. Особливості `bg`

- Процеси можуть продовжувати виводити дані у термінал, навіть у фоні.
- Це може заважати роботі — для уникнення перенаправляй вивід:

```bash
python3 app.py > app.log 2>&1 &
```

- Не всі програми коректно працюють у фоні (наприклад, `top`, `htop`, `mc`).
- Якщо закрити термінал, процеси, запущені через `bg`, завершаться (отримають сигнал SIGHUP).
- Для збереження процесу після закриття терміналу використовують `nohup` або `disown`.

---

## 7. Обмеження

- `bg` працює **тільки** з процесами, які були зупинені через `Ctrl+Z`.
- Неможливо перевести у фон процес, який стартував у передньому плані без зупинки.
- Якщо процес не реагує на `bg`, можливо, це інтерактивна програма, що не підтримує роботу у фоні.

---

## 8. Типові помилки

| Помилка | Причина | Рішення |
|-|-|-|
| `bg` не запускає процес | Програма не підтримує роботу у фоні | Використовувати інші інструменти або запускати з `&` |
| Вивід процесу засмічує консоль | Відсутнє перенаправлення виводу | Перенаправляти stdout і stderr у файл |
| Процес завершується при закритті терміналу | Процес отримує сигнал SIGHUP | Використовувати `nohup` або `disown` |

Дякую за детальний виклад про команду `fg` у bash!

Ось коротке резюме і ключові моменти, які ти описав:

---

## Команда `fg` — основне

- Повертає зупинений (`Ctrl+Z`) або запущений у фоні (`bg`) процес у передній план.
- Дозволяє взаємодіяти з процесом через термінал (читати вивід, вводити дані).
- Блокує shell, поки процес активний або не буде зупинений.

---

## Основні кроки при роботі з `fg`:

1. Запускаєш процес.
2. Зупиняєш його `Ctrl+Z` або запускаєш у фоні (`&`).
3. Перевіряєш список задач командою `jobs`.
4. Повертаєш потрібну задачу у передній план командою `fg` або `fg %номер`.
5. Взаємодієш з процесом, або завершуєш його `Ctrl+C`.

---

## Важливі нюанси:

- `fg` працює лише з процесами, запущеними у поточному shell.
- Якщо процес виводить багато інформації — термінал буде зайнятий цим виводом.
- Деякі CLI-програми (редактори, `top`, `less`) можуть некоректно відновлюватись після Ctrl+Z → fg.
- При потребі можна переглянути PID процесів через `jobs -l`.
- Процеси, запущені з `&`, також можна повернути у foreground через `fg`.
- Якщо процес зависає, краще його перезапустити.

---

## Практичні сценарії для розробників:

- Локальний сервер (PHP, Node.js, Python) — Ctrl+Z → bg → fg для управління.
- Редагування файлів у nano/vim — Ctrl+Z → fg для повернення.
- CLI-інструменти (MySQL, psql, redis-cli) — випадковий Ctrl+Z → fg для відновлення.
- Взаємодія з інтерактивними shell-процесами.

---

## Обмеження:

- Не можна повернути процес з іншої сесії чи терміналу.
- Процеси не збережуться після закриття терміналу (для цього треба `nohup` або `screen`, `tmux`).

## Що таке `nohup`

- `nohup` — команда, що запускає процес, ігноруючи сигнал **SIGHUP** (hangup), який надсилається при закритті терміналу або розриві SSH-сесії.
- Завдяки цьому процес продовжує працювати навіть після виходу з сесії.
- Вивід процесу за замовчуванням пишеться у файл `nohup.out`, якщо не вказано інше.

---

## Основні моменти використання

```bash
nohup command > output.log 2>&1 &
```

- `command` — команда, яку потрібно запустити.
- `> output.log` — перенаправляє стандартний вивід у файл.
- `2>&1` — перенаправляє помилки у той самий файл.
- `&` — запускає процес у фоні.

---

## Чому `nohup` потрібен

- `bg`, `fg`, `Ctrl+Z` — працюють лише в межах поточної сесії, процеси вмирають при закритті терміналу.
- `nohup` — дозволяє процесу жити поза межами сесії.
- Для повноцінного керування сесіями краще використовувати `tmux` або `screen`.

---

## Коли використовувати `nohup`

- Запуск довготривалих скриптів (імпорт, ETL, тренування моделей).
- Запуск серверів, ботів, демо-скриптів, які мають працювати безперервно.
- Автоматичні завдання, які не потребують інтерактивної взаємодії.

---

## Як перевірити статус процесу

- `ps aux | grep command` — знайти процес у системі.
- Перегляд логів через `tail -f output.log`.
- `jobs` працює лише у поточній сесії, після виходу процеси не відображаються.

---

## Поширені помилки

- Забути `&` — процес запускається у передньому плані і блокує термінал.
- Процес чекає на ввід — `nohup` не подбає про це, потрібна безінтерактивна команда.
- Лог-файли ростуть без контролю — треба налаштовувати ротацію логів.

---

## Важлива відмінність від `bg/fg`

| Команда  | Працює після закриття терміналу? | Керування процесом із терміналу |
|----------|---------------------------------|---------------------------------|
| `bg`     | Ні                              | Часткове (поки сесія жива)      |
| `fg`     | Ні                              | Повне (поки сесія жива)         |
| `nohup`  | Так                             | Немає (після виходу)             |
| `tmux`/`screen` | Так                       | Повне                           |

# Демонстрація роботи з `nohup` — послідовність команд

---

## 1. Підготовка: створимо тестовий скрипт, який працюватиме довго

```bash
cat << 'EOF' > long_script.sh
#!/bin/bash
for i in {1..30}
do
  echo "Iteration $i: $(date)"
  sleep 2
done
EOF

chmod +x long_script.sh
```

---

## 2. Запуск скрипта з `nohup` у фоновому режимі (без перенаправлення логів)

```bash
nohup ./long_script.sh &
```

Очікуємо повідомлення:

```
nohup: ignoring input and appending output to 'nohup.out'
```

---

## 3. Перевірка, що процес запущений

```bash
ps aux | grep long_script.sh | grep -v grep
```

---

## 4. Перегляд логів роботи процесу (вивід за замовчуванням у `nohup.out`)

```bash
tail -f nohup.out
```

---

## 5. Запуск скрипта з `nohup` і перенаправленням логів у окремий файл

```bash
nohup ./long_script.sh > mylog.log 2>&1 &
```

---

## 6. Перевірка логів у реальному часі

```bash
tail -f mylog.log
```

---

## 7. Демонстрація типових помилок

### 7.1. Запуск без `&` — процес блокує термінал

```bash
nohup ./long_script.sh
# (натисніть Ctrl+C для переривання)
```

Пояснення: процес працюватиме у передньому плані, термінал буде заблокований.

---

### 7.2. Запуск без `nohup` — процес завершується після виходу з сесії

```bash
./long_script.sh &
exit
```

Після повторного підключення перевірте, що процес зупинився:

```bash
ps aux | grep long_script.sh | grep -v grep
```

---

## 8. Перевірка `jobs` у поточній сесії

```bash
jobs
```

Після виходу з сесії команда `jobs` не покаже процеси.

---

## 9. Завершення процесу (припустимо, що знаємо PID)

```bash
ps aux | grep long_script.sh | grep -v grep
kill <PID>
```

---

## 10. Очищення логів (щоб логи не росли безмежно)

```bash
> nohup.out
> mylog.log
```

# Додатково (опціонально):

- Запуск `nohup`-процесу з імітацією помилки (наприклад, команда, що чекає вводу).
- Демонстрація запуску `nohup` з `disown` (щоб відв’язати процес від shell).
- Короткий огляд `tmux` або `screen` як кращих інструментів для керування довготривалими сесіями.
