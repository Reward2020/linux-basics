
# 🚀 Однокнопковий Деплой Веб-додатка у VirtualBox (Ubuntu 24.04 LTS)

Проєкт демонструє сучасний DevOps-підхід до автоматизації розгортання контейнеризованих додатків (Infrastructure as Code) на віддалений сервер через SSH за допомогою модульних Bash-скриптів та Docker Compose.  
Як демонстраційний додаток використовується розгортання простого веб-сайту (файли з `game_code/index.html`) у контейнері Docker на віддаленій віртуальній машині через SSH.
Використовується модульний підхід із Bash-скриптами та Docker Compose.

---

## 🏗️ Архітектура та структура проєкту

Проєкт повністю модульний. Усі конфігурації та логіка розділені на окремі файли на локальній машині:

```
demo/
├── README.md           # Цей файл з інструкціями
├── deploy.sh           # Головний скрипт-оркестратор (запускається локально)
├── prepare_step1.sh    # Перевірка та інсталяція Docker Compose на ВМ
├── prepare_step2.sh    # Підготовка папок і прав у /opt на ВМ
├── docker-compose.yml  # Конфігурація Docker Compose для запуску веб-сайту
└── game_code/          # Папка з вашим веб-кодом (index.html також можуть бути інші файли)
    └── index.html
```

---

## 🛠️ Передумови для запуску (Prerequisites)

Перед початком переконайтеся, що виконані такі умови:

1. **Доступ по SSH:**  
   На локальній машині налаштовано аліас `server`. Перевірити підключення можна командою:
   ```bash
   ssh server
   ```
2. **Права користувача:**  
   Користувач `<user_name>` на віртуальній машині має мати право виконувати команди `sudo`. Для повної автоматизації бажано налаштувати виконання команд `sudo` без запиту пароля (див. наступний розділ).
3. **Налаштування мережі VirtualBox:**  
   У налаштуваннях ВМ (Мережа -> Додатково -> Перенаправлення портів) має бути прокинуто порт для гри:  
   `Host Port: 8082 -> Guest Port: 8082`.

## 🔐 Налаштування sudo без пароля для користувача `<user_name>`

Для автоматизації деплою потрібно, щоб `<user_name>` міг виконувати команди з `sudo` без запиту пароля.

### Як це зробити:

Виконайте на віртуальній машині (через SSH) одну команду:

```bash
echo "<user_name> ALL=(ALL) NOPASSWD:ALL" | sudo tee /etc/sudoers.d/<user_name>-nopasswd && sudo chmod 0440 /etc/sudoers.d/<user_name>-nopasswd
```

- Ця команда створить файл `/etc/sudoers.d/<user_name>-nopasswd` з правилом, яке дозволяє `<user_name>` виконувати `sudo` без пароля.
- Права `0440` гарантують безпеку та коректне сприйняття файлу системою.

---

### Як перевірити, що налаштування працює:

1. Скиньте кеш пароля sudo:
   ```bash
   sudo -k
   ```
2. Спробуйте виконати команду з sudo:
   ```bash
   sudo whoami
   ```
3. Якщо команда виведе `root` без запиту пароля — налаштування успішне.

## 🧹 Додаткові кроки (за потреби)

### Видалення snap-версії Docker (якщо встановлена):

```bash
sudo snap remove --purge docker
```

### Встановлення офіційного Docker та Docker Compose через apt:

```bash
sudo apt update && sudo apt install -y docker.io docker-compose-v2
```

### Додавання користувача `<user_name>` до групи docker:

```bash
sudo usermod -aG docker <user_name>
```

### Перезавантаження віртуальної машини:

```bash
sudo reboot
```

## 🚀 Інструкція із запуску деплою

Вся демонстрація виконується однією командою з вашої локальної машини.

1. Перейдіть у папку з проєктом:

   ```bash
   cd demo
   ```

2. Надайте головному скрипту права на виконання:

   ```bash
   chmod +x deploy.sh
   ```

3. Запустіть процес деплою:

   ```bash
   ./deploy.sh
   ```

## 🔍 Опис роботи кроків автоматизації

### Крок 1: `prepare_step1.sh`

Перевіряє, чи встановлено сучасний плагін Docker Compose. Якщо команда відсутня, скрипт автоматично оновлює кеш пакетів apt та інсталює `docker-compose-v2` з офіційних репозиторіїв Ubuntu.

### Крок 2: `prepare_step2.sh`

Створює ізольовану директорію для додатка в `/opt/`. Передає права на неї користувачу `<user_name>`, гарантуючи безпечну роботу контейнерів.

### Крок 3: `docker-compose.yml`

Декларативний маніфест. Використовує готовий та легкий `Docker`-образ `nginx:alpine`. Перенаправляє зовнішній порт сервера `8082` на внутрішній порт контейнера `80` (`Nginx`), а також забезпечує автоматичний перезапуск додатка у разі збою системи (`restart: always`).

### Крок 4: Оркестрація (`deploy.sh`)

Послідовно запускає всі етапи, створює фінальний файл конфігурації на сервері через потокове введення (cat > ...) та піднімає сервіс у фоновому режимі (`-d`), видаляючи застарілі або осиротілі контейнери (`--remove-orphans`).

## 💡 Що саме ми демонструємо (DevOps Best Practices)

- **Модульність (Clean Code):** Логіку налаштування ОС (`prepare_*.sh`) та конфігурацію додатка (`docker-compose.yml`) винесено в окремі файли. Головний скрипт `deploy.sh` лише керує чергою виконання.
- **Потокове виконання через SSH:** Скрипти виконуються на віддаленому сервері прямо з пам'яті локальної машини. Не потрібно копіювати тимчасові файли через `scp`.
- **Параметризація через `bash -s`:** У кроці №2 ми передаємо локальний скрипт на сервер разом із динамічною змінною `"$APP_DIR"`. Це показує, як створювати гнучкі скрипти без жорстко прописаних шляхів.
- **Виробничий стандарт (FHS):** Додаток розгортається у директорії `/opt/my-app`, яка є стандартом Linux для стороннього ПЗ.
- **Безпека (Security):** Скрипт автоматично виправляє власника папки (`chown -R <user_name>:<user_name>`), що дозволяє Docker-демону працювати з файлами без необхідності запускати додаток від імені root.
