# Лекція 16: Практичні сценарії роботи з логами на Ubuntu 22.04

---

## Частина 1: Стандартна ієрархія логів

### Сценарій 1: Огляд каталогу /var/log

```bash
# Переглянути вміст каталогу з логами
ls -lh /var/log

# Подивитись конкретний системний лог, наприклад syslog
head -n 20 /var/log/syslog

# Перевірити права доступу до логів
ls -l /var/log/syslog
```

**Пояснення:**  
- Логи зберігаються у /var/log, бо це стандартне місце для логів у Linux.  
- Файли syslog, auth.log, kern.log — глобальні системні логи.  
- Права доступу обмежують, хто може читати логи (зазвичай root або користувачі з групи adm).

---

### Сценарій 2: Логи вебсервера Nginx

1. Встановити nginx (якщо ще не встановлений):

```bash
sudo apt update
sudo apt install nginx -y
```

2. Перевірити логи Nginx:

```bash
ls -lh /var/log/nginx/
head -n 20 /var/log/nginx/access.log
head -n 20 /var/log/nginx/error.log
```

3. Зробити тестовий запит і подивитися, як він відобразився в логах:

```bash
curl http://localhost
tail -n 10 /var/log/nginx/access.log
```

---

### Сценарій 3: Логи бази даних (наприклад, PostgreSQL)

1. Встановити PostgreSQL:

```bash
sudo apt install postgresql -y
```

2. Переглянути лог PostgreSQL:

```bash
sudo ls -lh /var/log/postgresql/
sudo tail -n 20 /var/log/postgresql/postgresql-*.log
```

## Частина 2: Інструменти аналізу логів

### Сценарій 4: Використання `tail` і `tail -f`

```bash
# Переглянути останні 10 рядків системного логу
sudo tail /var/log/syslog

# Моніторинг логу в реальному часі (в іншому терміналі)
sudo tail -f /var/log/syslog
```

Під час моніторингу можна запускати інші команди, наприклад:

```bash
logger "Test message $(date)"
```

і побачити, як з’являється запис у реальному часі.

---

### Сценарій 5: Використання `less` для безпечного читання

```bash
sudo less /var/log/syslog
```

- Навігація: стрілки, PageUp/PageDown, `/шаблон` — пошук, `q` — вихід

---

### Сценарій 6: Фільтрація за допомогою `grep`

```bash
# Знайти всі помилки в системному логу
sudo grep -i error /var/log/syslog

# Знайти всі записи, пов'язані з ssh
sudo grep ssh /var/log/auth.log
```

---

### Сценарій 7: Використання Pipes (|)

```bash
# Знайти всі помилки і показати останні 5
sudo grep -i error /var/log/syslog | tail -n 5

# Показати записи ssh і відфільтрувати за датою (наприклад, 2024-06-01)
sudo grep ssh /var/log/auth.log | grep "2026-05-01"
```

## Частина 3: Systemd Journal

### Сценарій 8: Перегляд журналу systemd

```bash
# Показати всі логи
journalctl

# Показати логи конкретного сервісу, наприклад nginx
sudo journalctl -u nginx

# Показати логи за останній запуск системи
journalctl -b

# Моніторинг в реальному часі
sudo journalctl -f
```

---

### Сценарій 9: Кореляція подій

```bash
# Переглянути логи nginx і ssh одночасно (через два термінали або у скрипті)
sudo journalctl -u nginx -f
sudo journalctl -u ssh -f
```

## Частина 4: Ротація логів

### Сценарій 10: Перегляд конфігурації logrotate

```bash
# Переглянути глобальний конфіг файл
cat /etc/logrotate.conf

# Переглянути конфіг для nginx
cat /etc/logrotate.d/nginx
```

---

### Сценарій 11: Примусова ротація логів

```bash
# Імітуємо ротацію для nginx
sudo logrotate -f /etc/logrotate.d/nginx

# Перевірити, що старі логи стиснуті і з'явився новий лог
ls -lh /var/log/nginx/
```

---

### Сценарій 12: Надсилання сигналу HUP для перезапуску логування

```bash
# Надіслати сигнал HUP nginx, щоб він відкрив нові лог-файли
sudo kill -HUP $(pidof nginx)
```

## Частина 5: Найкращі практики

### Сценарій 13: Логування через stdout/stderr (на прикладі простого скрипта)

Створити скрипт `logger.sh`:

```bash
#!/bin/bash
echo "INFO: Script started"
echo "WARNING: Something might be wrong" >&2
echo "INFO: Script finished"
```

Запустити і перенаправити в лог-файл:

```bash
bash logger.sh > script.log 2> script.err
cat script.log
cat script.err
```

### Сценарій 14: Виявлення Brute-force атак у логах ssh

```bash
sudo grep "Failed password" /var/log/auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -nr | head -10
```

Ця команда покаже IP-адреси з найбільшою кількістю невдалих спроб входу.

1. **`sudo grep "Failed password" /var/log/auth.log`**  
   Шукає в файлі `/var/log/auth.log` всі рядки, які містять фразу `"Failed password"`.  
   Це рядки, які повідомляють про невдалу спробу аутентифікації (наприклад, невдалий вхід по SSH).  
   `sudo` потрібен, бо файл може бути доступний лише для root.

2. **`awk '{print $(NF-3)}'`**  
   `awk` обробляє кожен рядок, розбиваючи його на поля за пробілами.  
   `NF` — це кількість полів у рядку.  
   `$(NF-3)` — це четверте поле з кінця.  
   
   У рядках типу:  
   ```
   May  6 12:00:00 server sshd[1234]: Failed password for invalid user root from 192.168.1.100 port 22 ssh2
   ```
   Поля з кінця будуть:  
   - $NF = ssh2  
   - $(NF-1) = 22  
   - $(NF-2) = 192.168.1.100  ← це IP-адреса  
   - $(NF-3) = from  
   
   Але тут видно, що IP — це $(NF-2), а не $(NF-3). Тож чому $(NF-3)? Можливо, у вашій системі формат трохи інший або ви хочете отримати поле перед IP. В стандартному випадку IP знаходиться у $(NF-3) чи $(NF-2) залежно від формату.  
   
   **Якщо IP знаходиться у $(NF-3), то ця команда виводить саме IP-адресу.**  
   
   Краще перевірити формат рядка, щоб упевнитися.

3. **`sort`**  
   Сортує всі IP-адреси в алфавітному (за замовчуванням) порядку, щоб однакові IP йшли підряд.

4. **`uniq -c`**  
   Підраховує кількість повторень кожного унікального IP (тобто скільки разів цей IP зустрічається у списку).

5. **`sort -nr`**  
   Сортує список за числом повторень у спадному порядку (`-n` — числовий, `-r` — зворотній порядок).

6. **`head -10`**  
   Виводить перші 10 рядків — топ-10 IP-адрес з найбільшою кількістю невдалих спроб входу.
