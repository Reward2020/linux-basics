
# Загальна мета скрипта `backup.sh`

Скрипт робить резервні копії певних таблиць у Google BigQuery для різних орендарів (tenant) залежно від вказаного середовища (наприклад, dev, qa, prod тощо). Він перевіряє наявність таблиць, кількість рядків у них і, якщо таблиця існує і не порожня, створює її клон з міткою часу.
## P.S:

Дані скрипт які описані в цьому документі можна обʼєднати в один так як вони досить схожі по своєму характеру та можна розділити аргументами `--migrate` та `--backup`

## Детальний розбір коду

### Оголошення змінних

```bash
ENV_LIST=$1 
copy_time=$(date +%Y%m%d)
tables="TABLE_name1 TABLE_name2 TABLE_name3 TABLE_name4 TABLE_name5"
```

- `ENV_LIST` — перший аргумент скрипта, в нього треба передати назву середовища (наприклад, `dev`, `qa`, `prod`).
- `copy_time` — поточна дата у форматі `YYYYMMDD`, використовується для імені резервних копій.
- `tables` — список імен таблиць, які потрібно обробити.

---

### Визначення списку орендарів (tenant) залежно від середовища

```bash
case $ENV_LIST in
  "dev")
    TENANTS_LIST="tenant1 tenant2 tenant3 tenant4 tenant5 tenant6 tenant7"
    ;;
  "qa")
    TENANTS_LIST="tenant1 tenant2 tenant3 tenant4 tenant5 tenant6 tenant7 tenant8 tenant9 tenant10"
    ;;
  "intg")
    TENANTS_LIST="tenant1 tenant2 tenant3 tenant4 tenant5"
    ;;
  "test")
    TENANTS_LIST="tenant1 tenant2 tenant3 tenant4 tenant5 tenant6 tenant7 tenant8 tenant9 tenant10 tenant11 tenant12 tenant13 tenant14 tenant15"
    ;;
  "prod")
    TENANTS_LIST="tenant1 tenant2 tenant3 tenant4 tenant5 tenant6 tenant7 tenant8 tenant9 "
    ;;
  *)
    echo "Invalid environment specified $ENV_LIST"
    exit 1
    ;;
esac
```

- Залежно від значення `$ENV_LIST` встановлюється відповідний список орендарів у змінну `TENANTS_LIST`.
- Якщо передано невідоме середовище — виводиться помилка і скрипт завершується.

---

### Основний цикл обробки

```bash
for ENV in $ENV_LIST; do
    for table in $tables; do
        for tenant_id in $TENANTS_LIST; do
            table_full="your-project-${ENV}:${tenant_id}_analytics.${table}"
            ...
        done
    done
done
```

- Цикл проходить по одному (або кількох, якщо передати кілька через пробіл) середовищу `$ENV_LIST`.
- Потім по кожній таблиці зі списку `tables`.
- Потім по кожному орендарю зі списку `TENANTS_LIST`.
- Формує повне ім’я таблиці у BigQuery у форматі:
  
  ```
  <your-project>-<ENV>:<tenant_id>_analytics.<table>
  ```
  
  Наприклад:
  
  ```
  <your-project>-dev:tenant1_analytics.TABLE_name1
  ```

---

### Перевірка існування таблиці

```bash
if bq show --format=prettyjson "$table_full" > /dev/null 2>&1; then
```

- Виконується команда `bq show` для таблиці.
- Якщо таблиця існує (команда виконується успішно), скрипт переходить до перевірки кількості рядків.
- Якщо таблиці немає — виводиться повідомлення і запис у файл `no-data-<ENV>.txt`.

---

### Отримання кількості рядків у таблиці

```bash
NumOfRaw=$(bq show --format=prettyjson "$table_full" | jq -r '.numRows // empty')
```

- Виконується команда `bq show`, вивід у форматі JSON.
- Через `jq` витягується поле `numRows` — кількість рядків у таблиці.
- Якщо не вдалося отримати число рядків (порожнє значення), виводиться повідомлення про помилку і запис у `no-data-<ENV>.txt`.
- Якщо кількість рядків 0 — виводиться повідомлення, що таблиця порожня, і запис у той же файл.

---

### Створення резервної копії таблиці

```bash
bq cp --clone --no_clobber "$table_full" "${table_full}_copy_${copy_time}"
```

- Якщо таблиця існує і не порожня, створюється клон таблиці з додаванням суфікса `_copy_<дата>`.
- Опції:
  - `--clone` — швидке клонування без копіювання даних (використовується внутрішній механізм BigQuery).
  - `--no_clobber` — не перезаписувати, якщо таблиця з таким ім’ям вже існує.
- Після успішного створення копії ім’я копії записується у файл `backups-<ENV>-prefix.txt`.
- Виводиться повідомлення про успішне створення резервної копії.

---

### Обробка випадків, коли таблиця не знайдена або порожня

- Якщо таблиця не існує — повідомлення і запис у `no-data-<ENV>.txt`.
- Якщо таблиця існує, але кількість рядків 0 або не вдалося отримати — теж повідомлення і запис у той же файл.

---

## Підсумок

- Скрипт приймає назву середовища (наприклад, `dev`).
- В залежності від середовища вибирає список орендарів.
- Для кожної таблиці і кожного орендаря формує повне ім’я таблиці у BigQuery.
- Перевіряє існування таблиці і кількість рядків.
- Якщо таблиця існує і не порожня — створює її клон з міткою часу.
- Логи про відсутність даних або помилки записує у файл `no-data-<ENV>.txt`.
- Логи про успішні бекапи — у файл `backups-<ENV>-prefix.txt`.

# Опис скрипта `migrate_table.sh`

Цей скрипт призначений для міграції (переносу) даних між середовищами (`dev`, `qa`, `prod` тощо) та орендарями (`tenant1`, `tenant2` і т.д.) у Google BigQuery. Він виконує запуск SQL-запитів, які підготовлені у файлах `.sql`, замінюючи в них шаблонні змінні на актуальні значення середовища, орендаря та дати.

---

## Покроковий розбір 

Початок скрипту аналогічний до попереднього:
- [Оголошення змінних](#оголошення-змінних)
- [Визначення списку орендарів (tenant) залежно від середовища](#визначення-списку-орендарів-tenant-залежно-від-середовища)

### Основні цикли

```bash
for ENV in $ENV_LIST; do
  for TENANT in $TENANTS_LIST; do
    for TABLE in $TABLE_LIST; do
      ...
    done
  done
done
```

- Проходимо по кожному середовищу.
- Для кожного середовища проходимо по кожному орендарю.
- Для кожного орендаря обробляємо кожну таблицю.

---

### Формування змінних для роботи з BigQuery

```bash
DATASET="your-project-${ENV}:${TENANT}_analytics"
COPY_TABLE="${TENANT}_analytics.${TABLE}_copy_${copy_time}"
```

- `DATASET` — повне ім’я датасету у BigQuery, наприклад `your-project-dev:tenant1_analytics`.
- `COPY_TABLE` — ім’я таблиці-копії, наприклад `tenant1_analytics.TABLE_name1_copy_20260512`.

---

### Перевірка існування датасету

```bash
if bq show "$DATASET" &>/dev/null; then
```

- Виконується перевірка, чи існує датасет у BigQuery.
- Якщо ні — виводиться повідомлення і пропускається обробка поточної таблиці.

---

### Перевірка існування таблиці-копії

```bash
if bq show --format=none "${DATASET%:*}:$COPY_TABLE" &>/dev/null; then
```

- Перевіряється, чи існує таблиця-копія.
- `${DATASET%:*}` — це видалення частини після двокрапки (тобто проекту), щоб отримати лише назву проекту (пояснення нижче).
- Якщо таблиця-копія існує — виконується вставка даних за допомогою SQL-запиту.
- Якщо ні — виводиться повідомлення і переходимо до наступної таблиці.

---

### Підготовка та запуск SQL-запиту

```bash
sed "s/<env>/${ENV}/g; s/<tenant>/${TENANT}/g; s/<current_date>/${copy_time}/g" ${TABLE}.sql > temp_${TABLE}_${TENANT}_${ENV}.sql
bq query --nouse_legacy_sql < temp_${TABLE}_${TENANT}_${ENV}.sql
rm temp_${TABLE}_${TENANT}_${ENV}.sql
```

- Використовується команда `sed` для заміни в шаблонному SQL-файлі (`TABLE.sql`) трьох плейсхолдерів:
  - `<env>` → поточне середовище (наприклад, `dev`)
  - `<tenant>` → поточний орендар (наприклад, `tenant1`)
  - `<current_date>` → дата у форматі `YYYYMMDD`
- Результат записується у тимчасовий файл `temp_<TABLE>_<TENANT>_<ENV>.sql`.
- Потім цей файл виконується як SQL-запит через `bq query`.
- Після виконання тимчасовий файл видаляється.

---

### Повідомлення про пропущені датасети або таблиці

- Якщо датасет не знайдено — повідомлення виводиться в консоль.
- Якщо таблиця-копія не знайдена — повідомляється про це і пропускається виконання запиту.

---

## Пояснення щодо `${DATASET%:*}`

- `DATASET` має формат `project:dataset`.
- `${DATASET%:*}` — це bash-розширення, яке видаляє найкоротший суфікс, що починається з `:`, тобто повертає тільки ім’я проекту.
- Це потрібно, бо `bq show` для таблиці очікує формат `project:dataset.table` або `dataset.table`.

---

## Підсумок

- Скрипт приймає середовище (`dev`, `qa` тощо).
- Визначає орендарів для цього середовища.
- Для кожного орендаря і кожної таблиці перевіряє наявність датасету і таблиці-копії.
- Якщо все є — генерує SQL-запит із заміною змінних і виконує його через `bq query`.
- Якщо датасет або таблиця відсутні — виводить відповідне повідомлення.
