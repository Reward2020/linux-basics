# Файл конфігурації SSH та підключення без пароля

---

## Частина 1: ⚙️ Конфігурація клієнта

### Проблема ручного введення

При підключенні по SSH зазвичай потрібно вручну вводити:

- Ім’я користувача
- IP-адресу або домен сервера
- Пароль або шлях до приватного ключа

**Приклад:**

```bash
ssh user@192.168.1.10
```

Потім система запитує пароль. Якщо працюєте з багатьма серверами — це дуже незручно і призводить до помилок (наприклад, неправильний логін, помилка в IP, забуття пароля).

---

### Анатомія конфігураційного файлу

- Файл конфігурації SSH клієнта розміщується за адресою:

  - Linux/macOS: `~/.ssh/config`
  - Windows (OpenSSH): `%USERPROFILE%\.ssh\config`

- Це текстовий файл, який дозволяє налаштовувати параметри підключення для різних хостів.

- Файл автоматично читається при запуску команди `ssh`.

---

### Ключові параметри блоку

Типовий блок у файлі виглядає так:

```plaintext
Host example
    HostName 192.168.1.10
    User user
    Port 22
    IdentityFile ~/.ssh/id_rsa
```

- **Host** — псевдонім або шаблон хоста
- **HostName** — реальна адреса сервера
- **User** — ім’я користувача для підключення
- **Port** — порт SSH (за замовчуванням 22)
- **IdentityFile** — шлях до приватного ключа
- **ProxyJump** — (пізніше) проксі-сервер для підключення через bastion

---

### Приклад мінімального конфігу

```plaintext
Host myserver
    HostName 192.168.1.10
    User alice
```

**Що це дає?**

Тепер можна підключатися просто так:

```bash
ssh myserver
```

і не вводити більше IP чи ім’я користувача.

---

### Керування декількома серверами

При підключенні до багатьох серверів варто:

- Використовувати унікальні псевдоніми `Host`
- Групувати параметри за середовищами (prod, dev, staging)
- Використовувати шаблони (про це далі)

**Приклад:**

```plaintext
Host prod-*
    User deploy
    IdentityFile ~/.ssh/prod_key

Host dev-*
    User devuser
    IdentityFile ~/.ssh/dev_key
```

Це допоможе уникнути конфліктів і спростить управління.

---

## Частина 2: 🎭 Шаблони та Wildcards

### Концепція Wildcard (*)

- `*` — це символ підстановки, що означає “будь-який рядок”
- Використовується у директиві `Host` для групування серверів

**Приклад:**

```plaintext
Host *.internal
    User internaluser
```

Це призначить параметри всім хостам, які закінчуються на `.internal`.

---

### Реальний сценарій: Внутрішня мережа

```plaintext
Host db.internal
    HostName 10.0.0.10
    User dbadmin

Host api.internal
    HostName 10.0.0.20
    User apiuser

Host *.internal
    IdentityFile ~/.ssh/internal_key
```

Тепер команди:

```bash
ssh db.internal
ssh api.internal
```

працюють автоматично, без додаткових параметрів.

---

### Порядок має значення!

- Специфічні хости потрібно описувати **першими**
- Загальні шаблони — **в кінці**

```plaintext
Host db.internal
    User dbadmin

Host *.internal
    User internaluser
```

Якщо поміняти порядок — специфічні налаштування можуть бути перекриті загальними.

---

### Перевизначення параметрів

Можна задавати різні параметри для однакових хостів у різних середовищах:

```plaintext
Host prod-db
    HostName prod-db.internal
    User produser
    IdentityFile ~/.ssh/prod_key

Host dev-db
    HostName dev-db.internal
    User devuser
    IdentityFile ~/.ssh/dev_key
```

---

### Wildcard != DNS

- Wildcard в SSH config — це просто шаблон для імені хоста, **не DNS-запит**
- Якщо `Host *.internal`, SSH не робить DNS-запит з wildcard, а просто порівнює рядок

---

### Перевірка через ssh -G

Команда:

```bash
ssh -G myserver
```

виводить всі параметри, які будуть застосовані при підключенні до `myserver`. Це корисно для відлагодження конфігурації.

## Частина 3: 🔑 Робота з IdentityFile

### Проблема "вгадування" ключів

- За замовчуванням `ssh` намагається підключитись із усіма ключами з агента або з `~/.ssh/id_*`
- Це може призводити до помилок, якщо сервер обмежує кількість спроб аутентифікації

---

### Директива IdentityFile

- Вказує конкретний приватний ключ для певного хоста

```plaintext
IdentityFile ~/.ssh/project_key
```

- Підтримує змінні, наприклад: `IdentityFile ~/.ssh/%r_key` де `%r` — ім’я користувача

---

### IdentitiesOnly yes

- Параметр, який змушує ssh використовувати **тільки** ключі, вказані через `IdentityFile`, ігноруючи інші ключі агента

```plaintext
IdentitiesOnly yes
```

- Зазвичай ставиться в блоці `Host`

---

### Практика: Генерація ключів під проект

- Генеруйте окремі ключі для кожного проекту:

```bash
ssh-keygen -f ~/.ssh/project1_key
ssh-keygen -f ~/.ssh/project2_key
```

- Це дозволяє ізолювати доступ і зручно керувати правами

---

### Встановлення специфічного ключа

При вказівці `IdentityFile` і `IdentitiesOnly yes` гарантуєте, що для цього хоста використовуватиметься саме цей ключ.

---

### Приклад ізольованого доступу

```plaintext
Host project1-server
    HostName 192.168.1.100
    User alice
    IdentityFile ~/.ssh/project1_key
    IdentitiesOnly yes
```

## Частина 4: 🕵️ SSH Agent

### Проблема Passphrase

- Якщо приватний ключ захищений паролем (passphrase), потрібно вводити його щоразу при підключенні
- Це незручно, особливо при частих підключеннях

---

### Що таке SSH Agent?

- Програма, яка зберігає приватні ключі в пам’яті і під час підключення автоматично їх підставляє
- Дозволяє вводити passphrase один раз за сесію

---

### Перевірка та запуск

- Перевірити чи агент запущений:

```bash
echo $SSH_AUTH_SOCK
```

- Запустити агент:

```bash
eval "$(ssh-agent -s)"
```

---

### Додавання ключів: ssh-add

- Додавання ключа до агента:

```bash
ssh-add ~/.ssh/id_rsa
```

- Після цього не потрібно вводити пароль при підключенні

---

### Перегляд завантажених ключів

```bash
ssh-add -l
```

Показує список ключів, які зараз завантажені в агент.

---

### Нюанси безпеки агента

- Ключі в агенті доступні процесам, які мають доступ до `SSH_AUTH_SOCK`
- Якщо зловмисник отримає доступ до агента — зможе підключатися від вашого імені
- Рекомендується обережно ставитись до прав доступу та використовувати timeout для агента

---

## Частина 5: 🛡️ Hardening та ProxyJump

### Вимкнення паролів (Hardening)

- У файлі сервера `/etc/ssh/sshd_config` вимкнути аутентифікацію по паролю:

```plaintext
PasswordAuthentication no
ChallengeResponseAuthentication no
UsePAM no
```

- Це змушує використовувати лише ключі, що підвищує безпеку

---

### ProxyJump (Bastion host)

- Дозволяє підключатися до внутрішніх серверів через проміжний сервер (bastion)

```plaintext
Host internal-server
    HostName 10.0.0.5
    User alice
    ProxyJump bastion.example.com
```

- При підключенні ssh автоматично підключиться спочатку до bastion, потім до внутрішнього сервера

---

### KeepAlive: тримаємо сесію

- Щоб уникнути розривів сесії, додають:

```plaintext
ServerAliveInterval 60
ServerAliveCountMax 3
```

---

### Повний приклад професійного конфігу

```plaintext
Host bastion
    HostName bastion.example.com
    User deploy
    IdentityFile ~/.ssh/bastion_key
    IdentitiesOnly yes

Host internal-*
    User internaluser
    ProxyJump bastion
    IdentityFile ~/.ssh/internal_key
    ServerAliveInterval 60
    ServerAliveCountMax 3
    IdentitiesOnly yes
```
