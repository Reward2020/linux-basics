# Розбір скрипту rsync.sh 

### Що таке shebang?

- Це спеціальний рядок на початку скрипту, який починається з символів `#!` і вказує операційній системі, якою програмою (інтерпретатором) потрібно виконувати цей файл.
- У твоєму випадку `#!/bin/bash` означає, що скрипт потрібно запускати через інтерпретатор Bash, який знаходиться за шляхом `/bin/bash`.

---

### Як це виникло?

- Ідея shebang з'явилася в Unix-подібних системах для того, щоб зробити текстові скрипти виконуваними файлами.
- Раніше, щоб запустити скрипт, потрібно було явно викликати інтерпретатор, наприклад:  
  ```
  bash script.sh
  ```
- З появою shebang стало можливим запускати скрипт безпосередньо, якщо у нього встановлені права на виконання:  
  ```
  ./script.sh
  ```
- Операційна система при запуску файлу перевіряє перший рядок: якщо він починається з `#!`, вона читає шлях до інтерпретатора і запускає його з цим файлом як аргументом.

---

Цей рядок називається **"shebang"** (іноді також "hashbang" або "sha-bang").
```bash
#!/bin/bash
```
- Вказує, що скрипт потрібно виконувати через Bash.

---

```bash
CONFIG_DIR="$HOME/Documents/GOIT/rsync"
WATCH_DIR="$HOME/Documents/GOIT/rsync"
```
- Задаємо дві змінні з шляхом до локальної директорії, де лежать конфігураційні файли (`.conf`) та файли для синхронізації.
- Обидві вказують на одну й ту ж папку.

---

```bash
REMOTE_HOST="goit-server"    # ім'я з ssh config, ваш аліас
```
- Змінна з ім’ям віддаленого сервера (SSH alias), на який будемо копіювати файли.

---

```bash
REMOTE_HOME=$(ssh "$REMOTE_HOST" 'echo $HOME')
```
- Виконуємо SSH-команду на віддаленому сервері, щоб дізнатись домашню директорію користувача (значення `$HOME` на сервері).
- Зберігаємо результат у змінну `REMOTE_HOME`.

---

```bash
REMOTE_PATH="${REMOTE_HOME}/app"
```
- Визначаємо шлях для деплою на віддаленому сервері — підпапка `app` у домашній директорії.

---

```bash
echo "Віддалена домашня директорія: $REMOTE_HOME"
echo "Віддалений шлях для деплою: $REMOTE_PATH"
echo "============================================================="
```
- Виводимо інформацію для користувача про віддалені шляхи.

### Функція `generate_function_json`

```bash
generate_function_json() {
  local conf_file="$1"
```
- Функція приймає один аргумент — шлях до конфігураційного файлу `.conf`.

```bash
  local base_name=$(basename "$conf_file" .conf)
```
- Визначаємо базову назву файлу без розширення `.conf` (наприклад, `example.conf` → `example`).

```bash
  local output_json="${CONFIG_DIR}/function_${base_name}.json"
```
- Визначаємо ім’я вихідного JSON-файлу, який буде згенеровано (наприклад, `function_example.json`).

```bash
  declare -A env_vars
```
- Оголошуємо асоціативний масив `env_vars` для збереження пар ключ=значення з конфігурації.

```bash
  while IFS='=' read -r key value; do
    [[ -z "$key" || "$key" =~ ^# ]] && continue
    env_vars[$key]=$value
  done < "$conf_file"
```
- Читаємо файл рядок за рядком.
- Розділяємо по знаку `=`.
- Пропускаємо порожні рядки та коментарі (рядки, що починаються з `#`).
- Зберігаємо ключі та значення у масив `env_vars`.

```bash
  json_parts=()
```
- Ініціалізуємо масив для частин майбутнього JSON.

```bash
  for key in "${!env_vars[@]}"; do
```
- Ітеруємо по всіх ключах масиву.

```bash
    if [[ "$key" == "LABELS" ]]; then
      IFS=',' read -ra label_pairs <<< "${env_vars[LABELS]}"
      labels_json_parts=()
      for pair in "${label_pairs[@]}"; do
        IFS='=' read -ra kv <<< "$pair"
        k="${kv[0]}"
        v="${kv[1]}"
        labels_json_parts+=("\"$k\":\"$v\"")
      done
      labels_json="{$(IFS=,; echo "${labels_json_parts[*]}")}"
      json_parts+=("\"labels\": $labels_json")
```
- Якщо ключ `LABELS`, парсимо його значення як список пар `key=value`, розділених комами.
- Формуємо JSON-об’єкт для `labels` зі всіма цими парами.
- Додаємо цей об’єкт у масив частин JSON.

```bash
    else
      val="${env_vars[$key]}"
      json_parts+=("\"$key\": \"$val\"")
    fi
  done
```
- Для інших ключів додаємо у JSON прості пари `"ключ": "значення"`.

```bash
  json_content="{$(IFS=,; echo "${json_parts[*]}")}"
```
- Об’єднуємо всі частини у валідний JSON-об’єкт.

```bash
  echo "$json_content" > "$output_json"
```
- Записуємо JSON у відповідний файл.

```bash
  echo "Згенеровано $output_json:"
  cat "$output_json"
  echo
}
```
- Виводимо повідомлення про створення файлу та його вміст.

### Основна логіка скрипта

```bash
shopt -s nullglob
```
- Вмикаємо опцію Bash, щоб шаблон `*.conf` повертав порожній список, якщо файлів немає, а не сам шаблон.

```bash
conf_files=("$CONFIG_DIR"/*.conf)
```
- Збираємо всі `.conf` файли у масив `conf_files`.

```bash
if [ ${#conf_files[@]} -eq 0 ]; then
  echo "Файли конфігурації не знайдені"
  exit 1
fi
```
- Якщо файлів немає — повідомляємо і завершуємо скрипт з помилкою.

```bash
for conf_file in "${conf_files[@]}"; do
  generate_function_json "$conf_file"
done
```
- Для кожного конфігураційного файлу виконуємо функцію генерації JSON.

```bash
shopt -u nullglob
```
- Вимикаємо опцію `nullglob`, повертаючи поведінку Bash до стандартної.

```bash
echo "Копіюємо function_*.json на сервер..."
scp "$CONFIG_DIR"/function_*.json ${REMOTE_HOST}:${REMOTE_PATH}/
# Або можемо використати так
# rsync -av "$CONFIG_DIR"/function_*.json ${REMOTE_HOST}:${REMOTE_PATH}/
```
- Копіюємо згенеровані JSON-файли на віддалений сервер у папку деплою.
- Закоментований варіант — через `rsync` (альтернатива `scp`).

---

```bash
echo -e "\n--- Dry-run: Перевірка файлів для синхронізації ---"
rsync -av --dry-run --exclude='.git/' --exclude='node_modules/' "$WATCH_DIR/" ${REMOTE_HOST}:${REMOTE_PATH}/rsync/
```
- Виконуємо пробний (dry-run) запуск `rsync` для перевірки, які файли будуть синхронізовані.
- Виключаємо директорії `.git` і `node_modules`.
- Синхронізуємо вміст `WATCH_DIR` у віддалений каталог `${REMOTE_PATH}/rsync/`.

---

```bash
read -p "Виконати реальну синхронізацію? (y/N): " CONFIRM
```
- Запитуємо у користувача підтвердження для реального копіювання.

---

```bash
if [[ "$CONFIRM" =~ ^[Yy]$ ]]; then
  echo "Виконуємо деплой..."
  rsync -av --exclude='.git/' --exclude='node_modules/' "$WATCH_DIR/" ${REMOTE_HOST}:${REMOTE_PATH}/rsync/
  echo "Деплой завершено."
else
  echo "Деплой скасовано."
fi
```
- Якщо користувач відповів `y` або `Y`, виконуємо реальне копіювання через `rsync`.
- Інакше — повідомляємо про скасування.

## Простий спосіб передати змінні через позиційні параметри

В скрипті можна зробити так:

```bash
#!/bin/bash

# Приймаємо параметри з командного рядка або використовуємо значення за замовчуванням
REMOTE_HOST=${1:-goit-server}
CONFIG_DIR=${2:-"$HOME/Documents/GOIT/rsync"}
WATCH_DIR=${3:-"$HOME/Documents/GOIT/rsync"}

echo "REMOTE_HOST = $REMOTE_HOST"
echo "CONFIG_DIR = $CONFIG_DIR"
echo "WATCH_DIR = $WATCH_DIR"

# Далі йде ваша основна логіка скрипта, яка використовує ці змінні
```

---

## Як запускати

```bash
./rsync.sh goit-server /шлях/до/конфігів /шлях/до/папки_для_синхронізації
```

Якщо хочете використати значення за замовчуванням, можна передати менше параметрів або не передавати взагалі:

```bash
./rsync.sh
```

---

## Пояснення

- `$1` — перший параметр (наприклад, ім'я віддаленого хоста)
- `$2` — другий параметр (папка з конфігураціями)
- `$3` — третій параметр (папка для синхронізації)
- Якщо параметр не передано, використовується значення за замовчуванням (`:-`)

# Підсумок

- Скрипт генерує JSON з `.conf` файлів.
- Копіює ці JSON на сервер.
- Робить "сухий запуск" синхронізації всіх файлів з локальної папки на сервер.
- Запитує підтвердження користувача.
- Якщо підтверджено — виконує реальну синхронізацію файлів.
