
## 1. Універсальність скриптів: передача зовнішніх параметрів

### Проблема
Жорстке кодування шляхів, імен баз даних, користувачів тощо ускладнює повторне використання скрипту в різних середовищах.

### Як зроблено у скрипті
- Замість жорстких значень використовується змінна з дефолтним значенням, наприклад:

```bash
DB_USER="${DB_USER:-goit}"
DB_NAME="${DB_NAME:-mytestdb}"
```

- Це означає: якщо змінна оточення `DB_USER` не встановлена, буде використано `"goit"`.
- Крім того, через `getopts` можна передати параметри з командного рядка:

```bash
while getopts ":u:d:" opt; do
  case $opt in
    u) DB_USER="$OPTARG" ;;
    d) DB_NAME="$OPTARG" ;;
    *) usage ;;
  esac
done
shift $((OPTIND -1))
```

- Таким чином, користувач може викликати скрипт як:

```bash
./upd_insert_users.sh -u myuser -d mydb users.txt
```

- Це робить скрипт гнучким і придатним для різних сценаріїв.

## 2. Валідація вхідних даних

### Чому це важливо
- Неочікувані або порожні значення можуть призвести до помилок, некоректної роботи або навіть до виконання небезпечних команд.
- Валідація допомагає уникнути SQL-ін’єкцій, неправильного формату даних та логічних помилок.

### Як зроблено у скрипті

- Перевірка, що ім’я користувача не пусте і відповідає регулярному виразу:

```bash
is_valid_username() {
  local username="$1"
  [[ "$username" =~ ^[a-zA-Z0-9_-]{1,30}$ ]]
}
```

- Перевірка формату email:

```bash
is_valid_email() {
  local email="$1"
  [[ "$email" =~ ^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$ ]]
}
```

- Якщо дані не проходять перевірку — відбувається пропуск з повідомленням:

```bash
if ! is_valid_username "$username"; then
  echo "Invalid username '$username', skipping"
  continue
fi
```

- Також перевіряється, що username не порожній:

```bash
if [[ -z "$username" ]]; then
  echo "Empty username, skipping"
  continue
fi
```

- Така валідація захищає від порожніх або шкідливих значень.

## 3. Різниця між `$@` та `$*` та безпечна обробка списків

### Основна різниця

- `$*` — всі позиційні параметри як один рядок, наприклад:

```bash
"$*"
```

об’єднує всі параметри в один рядок, розділений першим символом з `IFS` (за замовчуванням — пробіл).

- `$@` — всі позиційні параметри як окремі слова, наприклад:

```bash
"$@"
```

подає кожен параметр окремо, зберігаючи їх розділення.

### Чому це важливо

- Якщо параметри містять пробіли, `$*` зливає їх в один рядок, що може призвести до помилок.
- `$@` у лапках дозволяє безпечно ітеруватися по списку аргументів, зберігаючи кожен параметр окремо.

### Приклад у циклі

```bash
for server in "$@"; do
  echo "Connecting to $server"
done
```

Це безпечно обробить список серверів, навіть якщо в іменах є пробіли.

## 4. Обробка помилок та використання `set -euo pipefail`

### Що робить кожен параметр:

- `set -e` — скрипт одразу завершується при першій помилці (команда з ненульовим кодом повернення).
- `set -u` — помилка при зверненні до необ’явленої змінної (уникаємо помилок через пусті змінні).
- `set -o pipefail` — якщо будь-яка команда в конвеєрі завершується з помилкою, весь конвеєр вважається помилковим.

### Чому це важливо

- Це робить скрипт більш надійним, оскільки помилки не будуть ігноруватися.
- Запобігає виконанню подальших дій при помилках, що може призвести до непередбачуваних наслідків.

## 5. Механізм `trap`, lock-файли та очищення

### Lock-файл

- Запобігає одночасному запуску скрипту, що може спричинити конфлікти (наприклад, одночасні вставки в базу):

```bash
LOCKFILE="/tmp/pg_insert.lock"

if [ -e "$LOCKFILE" ]; then
  echo "Script is already running!" >&2
  exit 2
fi
touch "$LOCKFILE"
```

- При завершенні скрипту lock-файл видаляється:

```bash
cleanup() {
  rm -f "$LOCKFILE"
}
trap cleanup EXIT
```

### Механізм `trap`

- `trap cleanup EXIT` означає: функція `cleanup` буде викликана при виході зі скрипту — як при нормальному завершенні, так і при помилках або перериванні.
- Це гарантує, що lock-файл буде видалено навіть якщо скрипт аварійно завершиться.

### Автоматичний відкат (rollback)

- У цьому скрипті безпосередньо не реалізовано rollback транзакцій.
- Але можна розширити скрипт, обгорнувши всі вставки у транзакцію:

```bash
psql -U "$DB_USER" -d "$DB_NAME" -v ON_ERROR_STOP=1 <<EOF
BEGIN;
-- вставки тут
COMMIT;
EOF
```

- Якщо буде помилка, транзакція не буде зафіксована, і база залишиться у попередньому стані.
