
## Загальний огляд

В обох джобах є подібні блоки коду, які виконують такі дії:

1. Видаляють деякі файли з директорії `./env` за допомогою `find` + `xargs rm`.
2. Виконують команду `gsutil rsync` для синхронізації директорії `env` з Google Cloud Storage бакетом.
3. Запитують URL сервісу через `gcloud run services describe`.
4. Логують URL.
5. (В `.sync_per_env_infra`) Виконують HTTP-запит `DELETE` для очищення кешу.

---

## Де відбувається синхронізація?

Синхронізація — це команда:

```bash
gsutil rsync ...
```

В обох джобах вона є, але з невеликою відмінністю.

---

### В `.dry_run_per_env_infra`:

```bash
gsutil rsync -r -d -n -c -x "^.*(?<!${SCOPE}.conf)$" env gs://${BUCKET}/env
```

- Опції:
  - `-r` — рекурсивно.
  - `-d` — видаляти файли у цільовій папці, яких немає в джерелі.
  - `-n` — **dry-run**, тобто лише показати, що буде зроблено, але нічого не змінювати.
  - `-c` — перевіряти контрольні суми.
  - `-x` — виключити файли за регулярним виразом.

Отже, тут синхронізація **не відбувається фізично**, а лише симулюється — це dry run.

---

### В `.sync_per_env_infra`:

```bash
gsutil rsync -r -d -c -x "^.*(?<!${SCOPE}.conf)$" env gs://${BUCKET}/env
```

- Відсутня опція `-n`.
- Тобто команда виконує реальну (фізичну) синхронізацію — копіює файли, видаляє зайві.

---

## Висновок

- В `.dry_run_per_env_infra` — команда `gsutil rsync` виконується в **dry-run режимі**, тобто лише показує, що буде зроблено, але нічого не змінює.
- В `.sync_per_env_infra` — команда виконується **справжня синхронізація** файлів між локальною директорією `env` і бакетом Google Cloud Storage.

---

## Додатково

Після синхронізації в `.sync_per_env_infra` виконуються ще дії з очищення кешу сервісу через HTTP-запит:

```bash
curl -m 5 -w "%{http_code}" -X DELETE  -H "Authorization: Bearer $(gcloud auth print-identity-token)" $CONFIG_SERVER_URL/cleanCache
```

Це означає, що після оновлення файлів у хмарі ви очищуєте кеш конфігураційного сервера, щоб зміни вступили в силу.
