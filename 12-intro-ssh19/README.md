Ви можете створити пару ключів і налаштувати доступ до сервера не заходячи на нього вручну, використовуючи лише термінал на вашому ПК. [1, 2] 
Ось як це зробити одним ланцюжком команд, залежно від вашої ситуації:
## Варіант 1: Якщо у вас є пароль до сервера (найшвидший)
Ви можете згенерувати ключ локально і відправити його на сервер однією командою.

1. Згенеруйте ключ (якщо ще не зробили):
```bash
ssh-keygen -t ed25519
```
(Натискайте Enter, щоб вибрати шлях за замовчуванням).
2. Відправте його на сервер (це "встановить" ключ без вашого входу в консоль сервера):
* Для Linux/macOS:
```bash
ssh-copy-id user@host
```
* Для Windows (PowerShell):
```bash
cat ~/.ssh/id_ed25519.pub | ssh user@host "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
   ```
   [3, 4, 5, 6] 
Після введення пароля ключ буде додано, і наступного разу сервер впустить вас автоматично.

## Варіант 2: Якщо ви хочете створити ключ саме НА сервері (віддалено)
Якщо ваша мета — щоб пара ключів з'явилася в файловій системі самого сервера, ви можете дати команду на генерацію через SSH, не переходячи в інтерактивний режим:
```bash
ssh user@host "ssh-keygen -t rsa -b 4096 -f ~/.ssh/id_rsa -N ''"
```
Ця команда підключиться, створить ключ без пароля (-N '') і одразу завершить сесію.
## Що робити, якщо доступу до сервера взагалі немає (забули пароль)?
Якщо ви не можете підключитися ні по паролю, ні по ключу, "віддалено з ПК" через термінал це зробити неможливо з міркувань безпеки. У такому разі:

* Використовуйте Recovery Console у панелі керування вашого хостинг-провайдера (наприклад, [DigitalOcean](https://www.digitalocean.com/), [Hetzner](https://www.hetzner.com/)).
* Там ви зможете зайти під root і вручну додати свій публічний ключ у файл ~/.ssh/authorized_keys. [2, 7]

За замовчуванням команда ssh-copy-id сама шукає найсвіжіший створений ключ у вашій папці ~/.ssh/.
Однак, якщо у вас кілька різних ключів або ви створили ключ під іншим ім'ям (не стандартне id_rsa або id_ed25519), вам потрібно вказати шлях за допомогою прапорця -i:
```bash
ssh-copy-id -i ~/.ssh/my_custom_key.pub user@host
```
## Коли варто додавати шлях:

   1. Якщо у вас багато ключів: щоб випадково не відправити на сервер не той ключ.
   2. Якщо ви вказали свою назву файлу при генерації через ssh-keygen.
   3. Для певності: явне вказання шляху до файлу з розширенням .pub гарантує, що все пройде коректно.

Важливий нюанс: ssh-copy-id копіює саме публічний ключ (той, що з .pub). На вашому ПК при цьому залишається приватний ключ, який і буде використовуватися для входу.


---

Якщо ви скопіювали публічний ключ на сервер, тепер для підключення за допомогою приватного ключа (того, що без розширення .pub) є два основні шляхи:
## 1. Автоматичний (якщо назва ключа стандартна)
Якщо ваш файл називається id_rsa або id_ed25519 і лежить у папці ~/.ssh/, SSH-клієнт знайде його сам. Вам достатньо просто ввести:
```bash
ssh user@host
```
## 2. Ручний (якщо ключ має іншу назву або шлях)
Використовуйте прапорець -i (identity file), щоб вказати шлях до приватного ключа:
```bash
ssh -i ~/.ssh/my_private_key user@host
```
Важливо: тут ми вказуємо файл БЕЗ .pub.
## 3. Налаштування "Config" (найзручніший спосіб)
Щоб кожного разу не писати довгі команди, створіть (або відредагуйте) файл config на вашому ПК у папці ~/.ssh/:

1. Відкрийте файл: nano ~/.ssh/config
2. Додайте туди такі рядки:
```plaintext
Host my-server
    HostName 1.2.3.4
    User your_username
    IdentityFile ~/.ssh/my_private_key
```
3. Тепер ви зможете заходити на сервер просто командою:
```bash
ssh my-server
```
   
------------------------------
Можлива помилка (Permissions 0644):
Якщо ви вручну переносили приватний ключ, система може видати помилку, що він "занадто відкритий". Це лікується командою (на Linux/macOS):
```bash
chmod 600 ~/.ssh/my_private_key
```
Це дасть право читання ключа тільки вашому користувачу.

## Копіювання файлів: 
Налаштування конфігурації автоматично підтягнуться і для утиліти scp:
```bash
scp file.txt my-server:/remote/dir/
```
Якщо ви захочете перевірити правильність налаштувань у режимі налагодження, скористайтеся командою ssh -v my-server.

## Створення ключа для іншого користувача

Як адміністратор, ви можете самостійно згенерувати пару ключів для такого користувача, налаштувати сервер, а потім передати йому приватний ключ. Оскільки у користувача немає пароля, це єдиний спосіб надати йому перший доступ.
Ось покроковий алгоритм дій на вашому сервері:
## 1. Перейдіть у профіль користувача
Увійдіть на сервер під своїм root або sudo акаунтом та перемкніться на профіль користувача (наприклад, з іменем clientuser):
```bash
sudo su - clientuser
```
## 2. Створіть директорію та згенеруйте ключі
Виконайте генерацію ключа від імені цього користувача, щоб файли одразу отримали правильного власника:
```bash
mkdir -p ~/.ssh
chmod 700 ~/.ssh
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519 -N ""
```
Прапорець `-N ""` створює ключ без парольного захисту (passphrase), щоб користувач міг підключатися автоматично.
У цій команді за назву файлу відповідає аргумент після прапорця `-f`. Текст `ed25519` у частині `-t ed25519` — це тип алгоритму шифрування (технологія ключа)
#### Ви можете змінити назву файлу наступним чином:
```bash
ssh-keygen -t ed25519 -f ~/.ssh/my_custom_name -N ""
```
## 3. Авторизуйте публічний ключ
Додайте згенерований публічний ключ до списку дозволених на сервері:
```bash
cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys
or
cat ~/.ssh/my_custom_name.pub >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
```
## 4. Виведіть приватний ключ та передайте його
Тепер вам потрібно скопіювати приватний ключ, щоб безпечно віддати його користувачу:
```bash
cat ~/.ssh/privet_key
```

* Скопіюйте весь текст (починаючи з `-----BEGIN OPENSSH PRIVATE KEY-----` до `-----END OPENSSH PRIVATE KEY-----`).
* Передайте його користувачу безпечним каналом зв'язку (наприклад, через зашифрований месенджер або менеджер паролів, але не відкритим текстом в email).
* Видаліть приватний ключ з сервера після копіювання задля безпеки: `rm ~/.ssh/privet_key`.

## 5. Перевірка налаштувань SSH-демона
Переконайтеся, що у файлі конфігурації SSH на сервері `/etc/ssh/sshd_config` увімкнено автентифікацію за ключами:
```bash
PubkeyAuthentication yes
```
Якщо ви змінювали цей файл, не забудьте перезапустити службу: `sudo systemctl restart ssh` або `sudo service ssh restart`.

## Що має зробити користувач на своїй машині?

1. Зберегти отриманий текст у файл (наприклад, `my_key.pem`).
2. Якщо користувач на `Linux/macOS`, він обов'язково має виставити правильні права на файл, інакше SSH клієнт видасть помилку безпеки:
```bash
chmod 400 my_key.pem
```
3. Підключитися до сервера, вказавши шлях до файлу:
```bash
ssh -i my_key.pem clientuser@server_ip
```
(Або налаштувати файл `~/.ssh/config`, як ми розглядали у попередньому кроці).

## P.S: Невеличкий контекст
Якщо ви не використовуєте алгоритм `ed25519`, утиліта `ssh-keygen` підтримує інші типи ключів за допомогою прапорця `-t`.
Ось усі доступні альтернативні варіанти, їх особливості та актуальний статус безпеки:
## 1. RSA (Найбільш сумісний)
Найстаріший та найпопулярніший алгоритм. Зараз його безпечність повністю залежить від довжини ключа (кількості біт).

* Команда: `ssh-keygen -t rsa -b 4096 -f ~/.ssh/my_rsa_key -N ""`
* Важливо: Обов'язково додавайте прапорець `-b 4096`. Ключі довжиною `2048` біт або менше сьогодні вважаються застарілими й небезпечними.
* Плюс: Працює абсолютно на всіх старих і нових серверах та мережевому обладнанні (комутатори, роутери Cisco/MikroTik).

## 2. ECDSA (Алгоритм на еліптичних кривих)
Сучасний алгоритм, який є стандартом безпеки у багатьох державних установах США (стандарт NIST).

* Команда: `ssh-keygen -t ecdsa -b 521 -f ~/.ssh/my_ecdsa_key -N ""`
* Важливо: Прапорець `-b` визначає розмір кривої. Доступні варіанти: `256, 384 або 521`. Рекомендується використовувати максимальний — `521`.
* Плюс: Ключі значно менші за розміром, ніж RSA, а процес шифрування/дешифрування відбувається набагато швидше.

## 3. Ed25519-SK та ECDSA-SK (Ключі безпеки / Апаратні токени)
Спеціальні модифікації алгоритмів для роботи з фізичними USB-токенами (наприклад, YubiKey).

* Команда: `ssh-keygen -t ed25519-sk -f ~/.ssh/my_hardware_key -N ""`
* Як це працює: Ключ генерується всередині апаратного пристрою. Щоб підключитися по SSH, користувач повинен фізично вставити USB-токен у свій комп'ютер.

------------------------------
## 🛑 Варіанти, які ЗАБОРОНЕНО використовувати:
Якщо ви побачите ці варіанти в старих інструкціях, ігноруйте їх:

* `-t dsa` (Digital Signature Algorithm) — повністю застарів, зламаний і вимкнений за замовчуванням у всіх сучасних версіях OpenSSH.
* `-t rsa1` — використовувався для першої версії протоколу SSHv1, яка містить критичні вразливості.

## 💡 Лайфхак: Генерація без прапорця -t
Якщо ви взагалі приберете -t з команди:

`ssh-keygen -f ~/.ssh/default_key -N ""`

Утиліта автоматично створить ключ того типу, який прописаний за замовчуванням у конфігурації `OpenSSH` вашої операційної системи. У сучасних дистрибутивах Linux це зазвичай і є `ed25519` або `rsa на 3072/4096` біт.


# P.S: Ці зміни потрібно робити саме на сервері.


## Практики конфігурації сервера, дії на сервері (/etc/ssh/sshd_config)
Редагуйте головний файл конфігурації демона SSH на сервері за допомогою sudo nano `/etc/ssh/sshd_config`. Внесіть такі зміни для захисту від brute-force атак:

* Вимкнення паролів: Повністю закриває можливість підбору паролів.
```plaintext
PasswordAuthentication no
PubkeyAuthentication yes
```
* Заборона входу для root: Зловмисник не зможе підключитися напряму під суперкористувачем. Спочатку заходьте під звичайним користувачем, а потім підвищуйте привілеї через sudo.
```plaintext
PermitRootLogin no
```
* Зміна стандартного порту: Зміна порту з 22 на будь-який інший (наприклад, 2222) рятує від 99% автоматизованих бот-сканерів.
```plaintext
Port 2222
```
* Обмеження користувачів: Дозволяйте доступ лише конкретним особам.
```plaintext
AllowUsers alex maria
```
* Вимкнення порожніх паролів:
```plaintext
PermitEmptyPasswords no
```
* Таймаут сесії: Автоматично закриває підключення, якщо користувач відійшов від ПК.
```plaintext
ClientAliveInterval 300
ClientAliveCountMax 2
```

Важливо: Перед перезапуском служби перевірте синтаксис конфігурації командою sudo sshd -t. Якщо помилок немає, застосуйте зміни: sudo systemctl restart ssh (або sshd). Не закривайте поточну сесію, поки не перевірите нове підключення в іншому вікні термінала.

Файл `/etc/ssh/sshd_config` (із літерою d наприкінці, що означає daemon / демон) керує поведінкою SSH-сервера, який приймає вхідні підключення.
Щоб безпечно внести та застосувати зміни, виконайте ці кроки безпосередньо на сервері:


1. Відкрийте файл через редактор з правами суперкористувача:
```bash
sudo nano /etc/ssh/sshd_config
```
2. Внесіть зміни (наприклад, змініть `PasswordAuthentication` на `no`).
3. Обов'язково перевірте конфігурацію на наявність синтаксичних помилок перед перезапуском:
```bash
sudo sshd -t
```
Якщо команда нічого не вивела — усе налаштовано правильно.
4. Перезапустіть службу, щоб застосувати налаштування:
```bash
sudo systemctl restart ssh
```
(На деяких дистрибутивах, як-от `CentOS/RHEL`, команда буде: `sudo systemctl restart sshd`)

⚠️Головне правило безпеки: Після перезапуску служби не закривайте поточне вікно термінала. Відкрийте нове паралельне вікно на своєму комп'ютері та спробуйте підключитися до сервера за новими правилами. Якщо ви припустилися помилки, у першому (все ще відкритому) вікні ви зможете легко повернути старі налаштування назад.
