# Лекція: Моніторинг використання ресурсів (CPU, RAM, диск, IO)  
*Платформа: Ubuntu у VirtualBox без додаткових інсталяцій*  

---

## Частина 1: ⚡ Моніторинг CPU  

### Філософія моніторингу CPU  
- **Ідея:** CPU — це перший індикатор "здоров’я" системи.  
- **Load Average:**  
  - Load ≈ кількість ядер — система працює збалансовано.  
  - Load > ядер — процеси чекають, можливе гальмування.  
  - Високий Load + великий iowait = проблема з диском, а не з CPU.  

---

### Практика:  
1. **Перегляд Load і CPU у real-time:**  
```bash
top -d 1
```
- Зверніть увагу на `load average` у верхній частині.  
- Поля `%us` (користувацькі процеси) і `%sy` (системні) — аналізуємо, хто вантажить CPU.

2. **Запуск CPU-стресу:**  
```bash
stress --cpu 2 --timeout 30
```
- Запускає 2 потоки, що вантажать CPU на 100% протягом 30 секунд.  
- Спостерігаємо, як зростає Load і CPU usage.

3. **Покращений моніторинг — htop:**  
```bash
htop
```
- Сортуємо процеси за CPU (F6), дивимось дерево процесів (F5).  
- Фільтруємо за користувачем (F4).  

---

### Best practices CPU:  
- Використовуйте `mpstat -P ALL 1` для детального моніторингу по ядрах.  
- pidstat допомагає відстежити конкретні процеси:  
```bash
pidstat -u -p <PID> 1
```
- Визначте, чи проблема в коді (високий %us) чи системі (високий %sy або iowait).

---

## Частина 2: 🧠 Моніторинг памʼяті  

### Linux Memory Model  
- Пам’ять буферизується і кешується — не плутайте used і available.  
- `free -h` — основна команда, звертайте увагу на `available`.

---

### Практика:  
1. **Перевірка пам’яті:**  
```bash
free -h
vmstat 1
```
- Аналізуємо поля `si` (swap in) і `so` (swap out) — якщо вони не нульові, починається активне підкачування.

2. **Запуск стресу на RAM:**  
```bash
stress --vm 1 --vm-bytes 1G --vm-keep --timeout 30
```
- Виділяє 1 ГБ RAM і утримує її.  
- Спостерігаємо, як падає `available` у `free -h`.  

3. **Знаходження "важких" процесів:**  
```bash
ps aux --sort=-%mem | head -n 10
```

---

### Best practices RAM:  
- Використовуйте `vmstat` для моніторингу пам’яті і swap.  
- Пам’ятайте про OOM-killer: щоб уникнути, ставте memory limits (особливо у Docker).  
- Визначайте реальне споживання через RES (resident memory) у `top` або `ps`.

---

## Частина 3: 💾 Моніторинг диска (Disk Usage)  

### Чому диск переповнюється?  
- Логи, кеші, тимчасові файли.  
- Inodes — не забувайте про ліміт на кількість файлів.  

---

### Практика:  
1. **Перевірка зайнятого місця:**  
```bash
df -h
df -i
```
- Якщо Inodes > 90%, це причина проблем.

2. **Пошук великих папок:**  
```bash
du -sh /var/log/*
du -sh * | sort -h
```

3. **Пошук великих файлів:**  
```bash
find / -type f -size +500M 2>/dev/null
```

4. **Моніторинг Docker (якщо є):**  
```bash
docker system df
docker system prune -f
```

---

### Best practices Disk:  
- Налаштуйте ротацію логів (`logrotate`).  
- Регулярно чистьте тимчасові файли.  
- Контролюйте Inodes і не створюйте мільйони дрібних файлів без необхідності.  

---

## Частина 4: 📊 Моніторинг IO  

### Ключові метрики:  
- Throughput (rkB/s, wkB/s)  
- IOPS  
- %util (завантаження диска)  
- await (час очікування операції)  

---

### Практика:  
1. **Інструмент iostat:**  
```bash
iostat -xz 1
```
- Спостерігаємо за %util і await.  

2. **Інструмент iotop (потрібні права root):**  
```bash
sudo iotop -o
```
- Визначаємо процеси з високим IO.

---

### Best practices IO:  
- Виносьте важкі IO операції у бекграунд.  
- Контролюйте iowait у `top` і await у `iostat`.  
- Використовуйте SSD для баз даних і критичних застосунків.

---

## Частина 5: 🚀 Практичні сценарії  

### Сценарій 1: Високе CPU (High Load)  
- Запускаємо:  
```bash
stress --cpu 4 --timeout 60
```
- Спостерігаємо в `top` і `htop` зростання CPU load.  
- Аналізуємо `mpstat -P ALL 1`.

### Сценарій 2: Брак RAM та Swap Thrashing  
- Запускаємо:  
```bash
stress --vm 2 --vm-bytes 2G --vm-keep --timeout 60
```
- Перевіряємо `free -h`, `vmstat 1`.  
- Спостерігаємо swap in/out.

### Сценарій 3: Дисковий Bottleneck  
- Імітуємо IO навантаження:  
```bash
dd if=/dev/zero of=/tmp/testfile bs=1M count=1024 oflag=direct
```
- Перевіряємо `iostat -xz 1` і `iotop`.

### Сценарій 4: "No space left on device"  
- Створюємо файл, що заповнює диск:  
```bash
fallocate -l 2G /tmp/bigfile
```
- Перевіряємо `df -h` і `df -i`.  
- Чистимо зайве.

## Практичні сценарії для VirtualBox з Ubuntu (без додаткових інсталяцій)

---

### Сценарій 1: Високе навантаження CPU (High Load)

**Мета:** показати, як виглядає високий load, як це відображається в `top`/`htop`, як працюють `mpstat` і `pidstat`.

**Що робити:**

1. Запустити кілька процесів, що навантажують CPU:

```bash
# Запускаємо 4 процеси, які навантажують CPU (підставити кількість ядер)
for i in {1..2}; do yes > /dev/null & done
```

2. Запустити `top` або `htop` (якщо встановлено) і подивитись на %CPU, load average, поля us, sy, iowait.

3. Виконати:

```bash
mpstat -P ALL 1
pidstat 1
```

4. Показати, що відбувається, якщо кількість процесів навантаження більша за кількість ядер.

5. Завершити процеси:

```bash
killall yes
```

---

### Сценарій 2: Брак пам'яті (RAM) і Swap Thrashing

**Мета:** продемонструвати, як система поводиться при нестачі RAM, як це видно у `free`, `vmstat`, `top`.

**Що робити:**

1. Запустити скрипт, що споживає багато пам’яті (bash-скрипт на алокацію пам’яті):

```bash
stress --vm 1 --vm-bytes 1G --vm-keep &
```
2. Відкрити новий термінал і виконати:

```bash
free -h
vmstat 1
top
```

3. Показати, як росте використання swap (si/so), як падає available.

4. Якщо swap відсутній або дуже маленький, показати, що відбувається (OOM killer).

5. Завершити процес:

```bash
killall stress
```

---

### Сценарій 3: Дисковий Bottleneck

**Мета:** показати навантаження на диск, iowait, як це видно у `iostat` та `iotop`.

**Що робити:**

1. Запустити інтенсивний запис на диск:

```bash
dd if=/dev/zero of=/tmp/testfile bs=1M count=1024 oflag=direct &
```

2. У іншому терміналі запускати:

```bash
iostat -xz 1
iotop -o
top
```

3. Показати, як росте %util, await, iowait.

4. Пояснити, що це означає для продуктивності системи.

5. Завершити `dd`:

```bash
killall dd
rm /tmp/testfile
```

---

### Сценарій 4: "No space left on device" (Заповнення диску)

**Мета:** показати, як перевірити заповнення диску, inodes, як знайти великі файли/папки.

**Що робити:**

1. Створити великий файл, наприклад:

```bash
fallocate -l 2G /tmp/largefile
```

2. Запустити:

```bash
df -h
df -i
du -sh /tmp/*
```

3. Показати, як знайти великі файли:

```bash
find /tmp -type f -size +500M
```

4. Пояснити, що робити, якщо диск або inodes заповнені.

5. Видалити файл:

```bash
rm /tmp/largefile
```

## Додаткові поради для лекції

- Для `htop` можна запропонувати встановити через `sudo apt install htop` (якщо дозволено).
- Пояснити, що всі інструменти (top, free, vmstat, iostat, pidstat, mpstat, iotop) входять в пакет `sysstat` або `util-linux`, який зазвичай встановлений за замовчуванням.
- Показати команди, які допомагають швидко знайти "важкі" процеси:

```bash
ps aux --sort=-%cpu | head -n 10
ps aux --sort=-%mem | head -n 10
```

- Пояснити, що в реальних системах часто використовують моніторинг через Prometheus+Grafana, але базові інструменти дають швидке розуміння стану системи.
